# 第2点：可视化分析 - 数据集操作

这一点只覆盖 `可视化分析-数据集操作`，不展开到后面的 `仪表板制作 / 交互分析 / 图表分析 / 查询控件 / 其他控件`，也不提前进入 `系统管理 / 权限管理`。
如果说上一点 `数据分析` 解决的是“选哪种分析终端”，那么这一点解决的就是“如何把原始数据变成可被分析终端直接消费的数据资产”。官方文档里，数据集被明确定位为“可视化分析的基础”，而且围绕它已经形成完整生命周期：`创建 -> 管理 -> 预览 -> 建模 -> 数据处理 -> 高级配置 -> 组合复用`。[创建数据集](https://help.aliyun.com/zh/quick-bi/user-guide/create-a-dataset)

## 点位定义
`数据集操作` 在 Quick BI 里本质上是“分析语义层”和“分析可用层”的建设过程。它不是简单导入一张表，而是把数据源表、自定义 SQL、逻辑关系、物理关系、字段语义、计算规则、加速策略和后续复用方式，统一收敛成一个可被仪表板、电子表格、大屏、即席分析等终端直接消费的数据对象。[创建数据集](https://help.aliyun.com/zh/quick-bi/user-guide/create-a-dataset) [管理数据集](https://help.aliyun.com/zh/quick-bi/user-guide/manage-dataset)

## 这个点解决什么业务问题
它主要解决 5 类问题。

- 把分散在不同数据表中的业务信息收敛成统一分析对象，避免每做一次报表都从源表重新拼接。[创建数据集](https://help.aliyun.com/zh/quick-bi/user-guide/create-a-dataset)
- 在分析前把模型关系、字段语义、计算方式先定义好，让业务分析不必直接面对底层库表和 SQL 细节。[关系模型概念介绍](https://help.aliyun.com/zh/quick-bi/user-guide/concept-introduction-understanding-the-relational-model) [数据处理概述](https://help.aliyun.com/zh/quick-bi/user-guide/data-processing-overview)
- 给数据资产增加组织、预览、复制、转让、批量管理等能力，使数据集成为可治理、可协作的资源，而不是一次性中间产物。[管理数据集](https://help.aliyun.com/zh/quick-bi/user-guide/manage-dataset) [管理数据集文件夹](https://help.aliyun.com/zh/quick-bi/user-guide/manage-data-set-folders)
- 在不改底层源表的前提下，对字段、计算、分组、展示格式等做分析向加工，缩短“从数据到分析”的距离。[数据处理概述](https://help.aliyun.com/zh/quick-bi/user-guide/data-processing-overview) [字段配置](https://help.aliyun.com/zh/quick-bi/user-guide/detail-processing)
- 当单一数据集不能满足灵活分析时，再通过“数据集组合”做报表侧跨数据集联动，而不必回到建模层反复重做宽表。[数据集组合](https://help.aliyun.com/zh/quick-bi/user-guide/data-set-combination)

## 核心功能能力
这一点建议按生命周期理解，而不是按零散页面理解。

- `创建`：Quick BI 支持通过可视化拖表或自定义 SQL 创建数据集，一个数据集里数据表或 SQL 数量最多 100 个。这说明它既支持业务友好式搭建，也支持偏开发者式建集。[创建数据集](https://help.aliyun.com/zh/quick-bi/user-guide/create-a-dataset)
- `组织与管理`：数据集支持搜索、筛选、协同授权、复制、重命名、转让、删除、批量转让、批量移动、批量清缓存、批量删除，已经是完整资源管理能力，不只是“保存一下”。[管理数据集](https://help.aliyun.com/zh/quick-bi/user-guide/manage-dataset)
- `文件夹治理`：数据集文件夹支持新建、重命名、删除，且支持 5 级目录，说明 Quick BI 默认把数据集当成长期积累的数据资产，而非单次分析草稿。[管理数据集文件夹](https://help.aliyun.com/zh/quick-bi/user-guide/manage-data-set-folders)
- `预览与检查`：有数据集使用权限的用户可以进入预览页，查看字段详情、预览前 100 行数据、查看模型结构，并从这里跳转到行列权限、加速配置、小 Q 问数等配置入口。[数据集预览页](https://help.aliyun.com/zh/quick-bi/user-guide/dataset-preview-page)
- `建模`：当前 Quick BI 的数据模型已经是双层架构，包含逻辑层和物理层。逻辑层用关系线建模，通常情况下优先在逻辑层建模；物理层则用传统关联/合并形成宽表节点。[关系模型概念介绍](https://help.aliyun.com/zh/quick-bi/user-guide/concept-introduction-understanding-the-relational-model)
- `关系模型`：官方强调关系模型的价值是简化建模、减少数据膨胀、提升复用性。它不要求用户先决定 Join 方式，只需配置关联字段，系统会在查询时自动处理更合适的关联和预聚合策略。[关系模型概念介绍](https://help.aliyun.com/zh/quick-bi/user-guide/concept-introduction-understanding-the-relational-model)
- `物理关联/合并`：物理层仍支持 LEFT / RIGHT / INNER / FULL JOIN，以及数据表合并，所以 Quick BI 并不是完全放弃传统宽表建模，而是同时保留更底层的建模能力。[关联、合并](https://help.aliyun.com/zh/quick-bi/user-guide/association-consolidation)
- `数据处理`：模型完成后，可以继续做计算字段、分组维度、字段配置等处理，把原始字段进一步整理成分析字段。[数据处理概述](https://help.aliyun.com/zh/quick-bi/user-guide/data-processing-overview)
- `字段配置`：字段层支持改显示名、备注、计算公式、字段类型切换、日期源格式解析、地理信息设置、图片类型设置等。这说明 Quick BI 数据集不只是结构层对象，也承担“字段语义治理”角色。[字段配置](https://help.aliyun.com/zh/quick-bi/user-guide/detail-processing)
- `高级配置`：数据集编辑页还能进入高级配置，对基础属性、加速配置、协同授权、行列权限和小 Q 问数做统一管理。[高级配置概述](https://help.aliyun.com/zh/quick-bi/user-guide/advanced-configuration-overview)
- `数据集组合`：当多个数据集需要按场景灵活联用时，可以在仪表板、大屏、电子表格中做“先聚合、后关联”的数据集组合，而不是先回建模层做固定关联。[数据集组合](https://help.aliyun.com/zh/quick-bi/user-guide/data-set-combination)

## 典型使用场景

- IT 或数据团队先把底层业务表做成标准数据集，再交给分析师和业务侧反复复用。[创建数据集](https://help.aliyun.com/zh/quick-bi/user-guide/create-a-dataset)
- 同一份底层数据要支撑多个分析场景时，用关系模型减少“大宽表复制粘贴式建模”，降低后续维护成本。[关系模型概念介绍](https://help.aliyun.com/zh/quick-bi/user-guide/concept-introduction-understanding-the-relational-model)
- 数据源表结构变化后，需要同步字段、清理失效字段、避免报表字段失效时，走数据集维护流程最合适。[管理数据集](https://help.aliyun.com/zh/quick-bi/user-guide/manage-dataset)
- 业务分析要用地图、日期、图片、计算指标等语义化字段时，在数据集层先做好字段定义，比在每张报表里重复配置更稳妥。[字段配置](https://help.aliyun.com/zh/quick-bi/user-guide/detail-processing)
- 需要跨多个独立数据集做灵活联动分析，但这些关系并不适合固化为单一宽表时，用数据集组合更合适。[数据集组合](https://help.aliyun.com/zh/quick-bi/user-guide/data-set-combination)

## 适用角色

- `空间开发者 / 数据开发者 / 数据分析师` 是这一点的核心操作者，因为创建、编辑、建模、字段处理、加速配置都偏资源生产侧。[管理数据集](https://help.aliyun.com/zh/quick-bi/user-guide/manage-dataset) [高级配置概述](https://help.aliyun.com/zh/quick-bi/user-guide/advanced-configuration-overview)
- `空间管理员 / 资源 Owner / 权限管理员` 更关注数据集授权、转让、目录治理和资源管理。[管理数据集](https://help.aliyun.com/zh/quick-bi/user-guide/manage-dataset) [管理数据集文件夹](https://help.aliyun.com/zh/quick-bi/user-guide/manage-data-set-folders)
- `分析人员` 也会参与使用权限范围内的数据集预览、使用和组合，但通常不会承担完整建模责任。[数据集预览页](https://help.aliyun.com/zh/quick-bi/user-guide/dataset-preview-page) [数据集组合](https://help.aliyun.com/zh/quick-bi/user-guide/data-set-combination)

## 关键使用路径
从官方资料看，这一节的标准路径可以稳定概括为：

1. 先连接数据源并创建数据集，可视化拖表或自定义 SQL 二选一。[创建数据集](https://help.aliyun.com/zh/quick-bi/user-guide/create-a-dataset)
2. 如果是多表分析，优先在逻辑层做关系建模；必要时进入物理层做关联或合并。[关系模型概念介绍](https://help.aliyun.com/zh/quick-bi/user-guide/concept-introduction-understanding-the-relational-model) [关系模型操作说明](https://help.aliyun.com/zh/quick-bi/user-guide/relational-model-operating-instructions) [关联、合并](https://help.aliyun.com/zh/quick-bi/user-guide/association-consolidation)
3. 模型完成后进入数据处理页，补充计算字段、分组维度、字段类型和展示格式等语义化配置。[数据处理概述](https://help.aliyun.com/zh/quick-bi/user-guide/data-processing-overview) [字段配置](https://help.aliyun.com/zh/quick-bi/user-guide/detail-processing)
4. 再进入高级配置，补充基础属性、加速、权限、小 Q 问数等增强项。[高级配置概述](https://help.aliyun.com/zh/quick-bi/user-guide/advanced-configuration-overview)
5. 之后通过数据集管理页做复制、转让、删除、批量治理，或进入分析终端直接消费。[管理数据集](https://help.aliyun.com/zh/quick-bi/user-guide/manage-dataset)
6. 如果报表层需要灵活跨数据集分析，再走数据集组合，而不是回头重建所有模型。[数据集组合](https://help.aliyun.com/zh/quick-bi/user-guide/data-set-combination)

## 前置条件 / 权限要求 / 版本限制

- 通用前提是先获取数据并连接数据源。[创建数据集](https://help.aliyun.com/zh/quick-bi/user-guide/create-a-dataset)
- 编辑数据集要求用户类型为开发者，且空间角色拥有 `数据集新建（编辑）权限` 和 `数据源使用权限`。[管理数据集](https://help.aliyun.com/zh/quick-bi/user-guide/manage-dataset)
- 数据集文件夹的新建、重命名、删除，仅空间管理员、空间开发者和具备数据集新建（编辑）权限的自定义角色可以操作。[管理数据集文件夹](https://help.aliyun.com/zh/quick-bi/user-guide/manage-data-set-folders)
- 数据集预览要求有 `数据集使用权限`。[数据集预览页](https://help.aliyun.com/zh/quick-bi/user-guide/dataset-preview-page)
- 物理关联/合并方面，`非同源数据表关联与合并仅专业版支持`，跨源还要求开启 Quick 引擎抽取加速；MySQL 暂不支持全关联；关联最多 5 层，合并最多 5 个表；API 数据源直连模式只支持单表建模。[关联、合并](https://help.aliyun.com/zh/quick-bi/user-guide/association-consolidation)
- 关系模型是 `V6.1` 引入的新能力。官方明确写了：V6.1 初版下，多逻辑表数据集在部分功能点上仍有限制，例如数据准备、复杂过滤条件相关能力、部分行级权限场景、仪表板复合查询控件、监控告警规则等。[关系模型概念介绍](https://help.aliyun.com/zh/quick-bi/user-guide/concept-introduction-understanding-the-relational-model)
- 小 Q 问数属于增值模块，需要额外购买。[高级配置概述](https://help.aliyun.com/zh/quick-bi/user-guide/advanced-configuration-overview)

## 与上下游点位的关系

- 它的上游是 `获取数据 / 连接数据源`。没有源数据，就没有数据集。[创建数据集](https://help.aliyun.com/zh/quick-bi/user-guide/create-a-dataset)
- 它是上一点 `数据分析` 的真正供给层。上一点里的仪表板、电子表格、大屏、即席分析、自助取数，本质上都建立在“可用数据集”之上。[创建数据集](https://help.aliyun.com/zh/quick-bi/user-guide/create-a-dataset)
- 它的下游就是后面的 `仪表板制作 / 交互分析 / 图表分析 / 查询控件 / 其他控件`。也就是说，后续章节更多解决“怎么展示和交互”，这一点解决“展示和交互之前，数据能不能正确、稳定、可复用”。[管理数据集](https://help.aliyun.com/zh/quick-bi/user-guide/manage-dataset)

## 限制与边界

- 这一点不是完整 ETL 平台介绍。Quick BI 的数据集操作支持一定建模与处理，但并不等于大规模离线数仓开发工具。[创建数据集](https://help.aliyun.com/zh/quick-bi/user-guide/create-a-dataset)
- 关系模型虽然更智能，但不是无代价。官方明确提示，相比传统物理关联，关系模型 SQL 更复杂，性能受数据量和模型复杂度影响更明显；如果查询性能要求极高、且你对数据分布掌控非常清楚，仍可考虑传统物理建模。[关系模型概念介绍](https://help.aliyun.com/zh/quick-bi/user-guide/concept-introduction-understanding-the-relational-model)
- 关系模型存在 `每次查询 1 万行聚合后结果限制`，而且无法靠分页拿到 1 万行之后的数据；如果要查更明细、维度枚举值极多的数据，可能需要改回物理建模。[关系模型概念介绍](https://help.aliyun.com/zh/quick-bi/user-guide/concept-introduction-understanding-the-relational-model)
- 数据集预览最多只看前 100 行，而且不支持直接导出。[数据集预览页](https://help.aliyun.com/zh/quick-bi/user-guide/dataset-preview-page)
- 数据集组合虽然灵活，但它不是万能替代。官方明确写明它只支持仪表板、大屏、电子表格，不支持自助取数和即席分析；并且会限制钻取、监控告警、数据集替换、波动分析、自动见解、智能问答，以及部分图表能力；展示最多 10000 条数据。[数据集组合](https://help.aliyun.com/zh/quick-bi/user-guide/data-set-combination)

## 售前 / 方案视角价值解读

- 从方案角度看，`数据集操作` 是 Quick BI 区别于“只会连库画图”的轻量 BI 工具的关键层。它提供了一个面向分析的语义层，让企业可以把“原始表”与“业务分析对象”分开管理。[创建数据集](https://help.aliyun.com/zh/quick-bi/user-guide/create-a-dataset)
- 从售前表述看，Quick BI 不只是报表产品，而是“带分析语义层的数据分析平台”。这一层使企业既能支撑统一口径报表，又能降低业务自助分析门槛。[关系模型概念介绍](https://help.aliyun.com/zh/quick-bi/user-guide/concept-introduction-understanding-the-relational-model)
- 如果客户常见痛点是“同一批底表反复造宽表、场景一变就得重建模型”，那么关系模型和数据集治理能力就是核心卖点，因为官方明确把它定位为减少数据膨胀、减少重复建模、提升复用性。[关系模型概念介绍](https://help.aliyun.com/zh/quick-bi/user-guide/concept-introduction-understanding-the-relational-model)
- 如果客户更关注大规模高性能固定分析场景，则需要更谨慎地讲：Quick BI 关系模型提高了灵活性，但复杂模型并不一定比传统宽表更快，必要时仍要结合物理建模与引擎加速设计。[关系模型概念介绍](https://help.aliyun.com/zh/quick-bi/user-guide/concept-introduction-understanding-the-relational-model) [高级配置概述](https://help.aliyun.com/zh/quick-bi/user-guide/advanced-configuration-overview)
- 如果客户希望“同一份数据资产既给 IT 做标准报表，又给业务做灵活分析”，这一点恰好是承上启下的支点：数据集把底层数据抽象成统一分析对象，后续不同终端再各自消费。[管理数据集](https://help.aliyun.com/zh/quick-bi/user-guide/manage-dataset)
