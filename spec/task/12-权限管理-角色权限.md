# 第12点：权限管理 - 角色权限

这一点只覆盖 `权限管理-角色权限`。
它和上一轮 `系统管理-权限管理` 的区别是：上一轮讲“权限体系怎么搭”，这一轮讲“角色到底怎么承载权限”。也就是把 Quick BI 的权限治理真正落到“谁扮演什么岗位、这个岗位能看到哪些模块、能执行哪些功能、该岗位和资源权限是什么关系”上。[角色管理及权限](https://help.aliyun.com/zh/quick-bi/user-guide/role-permissions) [角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)

## 点位定义
`角色权限` 本质上是 Quick BI 的“以岗赋权”机制。
Quick BI 不是直接把功能点零散发给每个人，而是先定义 `组织角色` 和 `空间角色`，再把用户或用户组挂到这些角色上。官方对这套能力的定位很明确：`以岗赋权、多岗多角色、统一管理、清晰化各职能边界`。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management) [自定义角色：以岗赋权，满足复杂多元的权限应用场景](https://help.aliyun.com/zh/quick-bi/product-overview/user-defined-roles)

## 这个点解决什么业务问题

- 把“岗位职责”转成系统里的权限边界，而不是每个用户单独配一堆散乱权限。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)
- 区分 `组织级职责` 和 `空间级职责`，避免平台超管、资源管理员、数据开发、业务分析、只读查看者都混在一个权限模型里。[角色管理及权限](https://help.aliyun.com/zh/quick-bi/user-guide/role-permissions)
- 支持一个人同时承担多个岗位，例如既是某空间的 ETL 工程师，又是组织级资源管理员。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)
- 在企业复杂协作场景下，把权限控制从“人治”变成“角色治理”，便于批量加人、换岗、交接和审计。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)

## 核心功能能力

- `双层角色体系`：Quick BI 把角色分成两层。
  - `组织角色`：作用于整个组织级能力。
  - `空间角色`：作用于某个工作空间内的能力边界。[角色管理及权限](https://help.aliyun.com/zh/quick-bi/user-guide/role-permissions)
- `预置组织角色`：官方预置 `组织管理员、权限管理员、普通用户` 三种组织角色。
  - 组织管理员负责组织信息、状态和成员管理。
  - 权限管理员负责后台统一分配和管理权限。
  - 普通用户是默认组织角色。[角色管理及权限](https://help.aliyun.com/zh/quick-bi/user-guide/role-permissions)
- `预置空间角色`：官方预置 `空间管理员、空间开发者、空间分析师、空间查看者` 四种空间角色。
  - 空间管理员：当前空间内权限最大，除全模块新建/编辑/使用/查看外，还可管理空间成员权限和作品。
  - 空间开发者：拥有所有模块的新建（编辑）、使用和查看权限。
  - 空间分析师：重点拥有数据门户、仪表板、数据大屏、电子表格、即席分析、自助取数、数据准备等新建/查看能力，以及数据集使用权限。
  - 空间查看者：拥有所有模块查看权限。[角色管理及权限](https://help.aliyun.com/zh/quick-bi/user-guide/role-permissions) [角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)
- `自定义角色`：专业版支持新增 `自定义组织角色` 和 `自定义空间角色`，可按企业岗位自定义功能权限范围。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management) [自定义角色：以岗赋权，满足复杂多元的权限应用场景](https://help.aliyun.com/zh/quick-bi/product-overview/user-defined-roles)
- `多岗多角色`：同一个用户可以被赋予多个角色，不需要把职责强行压缩到单一身份里。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)
- `批量关联能力`：角色支持直接添加用户，也支持切换到 `用户组` 进行跨组批量加人，还支持批量删除和批量变更角色。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)
- `功能权限配置`：角色权限的核心是定义功能模块范围，而不是逐个资源分配。
  - 组织角色可覆盖 `工作空间管理、企业安全、智能运维、外观配置、报表配置、地图配置、小Q问数、探索分析、订阅管理、自助取数、开放 API、数据服务、嵌入分析` 等组织级模块。
  - 空间角色可控制各空间模块的 `查看、新建（编辑）`，以及 `数据集 / 数据源使用权限`。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)
- `预置角色不可改`：官方预置组织角色和空间角色都不支持直接修改功能权限；如果预置角色不适合企业岗位边界，就需要通过自定义角色重建。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)
- `空间角色与文件夹管理关联`：拥有某模块新建（编辑）权限的用户，自动具备该模块文件夹的新建、重命名、删除能力；只有查看权限的用户不支持文件夹操作。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)
- `数据集使用权限的双通道生效`：官方明确说明，组织角色中的 `数据集使用` 权限与空间角色中的 `数据集使用` 权限“共同生效，有其一即可”。这是角色权限里比较关键的细节。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)

## 典型使用场景

- 给市场宣传人员创建“仅能做数据大屏”的空间角色，避免他看到和操作数据准备、仪表板、数据源等不相关模块。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)
- 给数据部门人员创建“ETL 工程师”空间角色，只开放数据准备、数据集、数据源相关能力，不给报表创作权限。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)
- 给组织级运维人员创建“智能运维工程师”组织角色，只开放智能运维能力，而不赋予过大的业务资源权限。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)
- 给负责资源投产的人员创建“资源管理员”组织角色，只开放资源包管理等组织级功能。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)
- 企业岗位职责复杂、人员兼职较多时，用多角色组合替代“一人一个大而全角色”。[自定义角色：以岗赋权，满足复杂多元的权限应用场景](https://help.aliyun.com/zh/quick-bi/product-overview/user-defined-roles)

## 适用角色

- `组织管理员`：是角色权限体系的设计者，也是唯一能配置自定义角色的人。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)
- `权限管理员`：更多是角色权限体系的执行者和日常治理者，但不一定负责自定义角色建模本身。[角色管理及权限](https://help.aliyun.com/zh/quick-bi/user-guide/role-permissions)
- `工作空间管理员`：负责空间成员角色落地，尤其是把用户绑定到具体空间角色。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)
- `业务负责人 / 方案负责人`：在实施期会参与定义“哪些岗位该拥有哪些模块权限”，因为这是角色设计的业务输入。

## 关键使用路径

1. 先区分当前要设计的是 `组织角色` 还是 `空间角色`。[角色管理及权限](https://help.aliyun.com/zh/quick-bi/user-guide/role-permissions)
2. 如果预置角色不满足岗位边界，再进入 `角色管理` 新建自定义角色。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)
3. 按岗位职责勾选功能权限：
   - 组织角色偏组织级模块。
   - 空间角色偏空间内模块的查看、新建（编辑）、使用。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)
4. 把用户或用户组加入该角色，必要时批量变更已有成员角色。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)
5. 对空间角色，最终还要在 `空间成员与信息` 页面把用户正式挂到对应空间，角色才真正落地到该空间。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)

## 前置条件 / 权限要求 / 版本限制

- `自定义角色` 仅专业版支持。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management) [自定义角色：以岗赋权，满足复杂多元的权限应用场景](https://help.aliyun.com/zh/quick-bi/product-overview/user-defined-roles)
- 只有 `组织管理员` 能配置自定义角色。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)
- 角色相关开放 API 当前只支持系统预置角色，暂不完整支持自定义角色。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)
- 预置组织角色和预置空间角色都 `不支持修改功能权限`；如需变更，只能新建自定义角色再替换。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)
- 新建组织角色时，系统会默认选中一部分企业应用和开放集成类权限；新建空间角色时，默认选中各模块展示权限。实施时不能默认它是“全空白模板”，需要检查默认勾选项。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)

## 与上下游点位的关系

- 它的上游是前面的 `系统管理-权限管理`，因为那一轮讲的是权限体系框架；这一点开始真正进入某一种权限治理对象。
- 它是后面 `资源权限` 的前置。角色权限先决定“用户能否进入某模块、是否具备某类功能能力”，资源权限再决定“具体哪张报表、哪个数据集、哪条数据能不能操作”。
- 它和 `集中授权` 有联系但不相同。角色权限偏“按岗位定义功能边界”，集中授权偏“按用户或资源批量发放具体资源权限”。

## 限制与边界

- `角色权限` 不等于 `资源权限`。官方明确把两者拆开：
  - 角色主要定义功能权限。
  - 资源权限则通过组织级、空间级集中资源授权来补充。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)
- 角色权限不是万能通行证。官方专门给了优先级规则：
  - 从功能角度，功能权限大于空间资源使用权限。
  - 从资源角度，功能权限又受到空间资源权限约束。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)
- 这意味着：
  - 如果空间角色把“仪表板模块”权限拿掉，即使用户原来对某些仪表板有资源权限，也会看不到仪表板入口。
  - 反过来，就算角色允许看某模块，也不代表能自动管理该空间内所有别人创建的资源，还要看资源层授权。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)
- 所以角色权限更像“功能边界框”，不是最终所有资源访问结果本身。

## 售前 / 方案视角价值解读

- 从售前角度，这一节最能说明 Quick BI 不是“粗粒度授权系统”，而是支持按岗位职责切分功能边界的企业级 BI 平台。
- 从方案设计看，自定义角色是非常重要的落地抓手。很多客户真正的痛点不是“有没有权限”，而是“默认角色太粗，岗位边界对不上”。Quick BI 把这一层显式做成了产品能力。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)
- 对客户最有说服力的一点是：Quick BI 支持 `多岗多角色`。现实企业里，一个人往往兼任多个职责，如果系统只能一人一角色，落地时很容易变形；而 Quick BI 在这点上更贴近真实组织结构。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)
- 另外，官方给出的“大屏管理员、ETL 工程师、资源管理员、智能运维工程师”这些示例，已经很接近企业实施中的真实岗位模板，适合直接拿来给客户讲角色设计思路。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)
