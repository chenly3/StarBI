# 第10点：系统管理 - 权限管理

这一点只覆盖 `系统管理-权限管理`。
它和后面专门的 `权限管理` 章节要分开理解：这一点讲的是 Quick BI 的“系统级权限框架”本身，也就是企业怎么把权限体系搭起来、开关怎么控、授权入口怎么定、审批怎么流转；后面那 5 个点才会分别细讲 `角色权限 / 资源权限 / 集中授权 / 数据行列权限 / 申请与审批权限` 的具体做法。官方权限管理总目录本身就是这样组织的。[权限管理](https://help.aliyun.com/zh/quick-bi/user-guide/permission-management/)

## 点位定义
`系统管理-权限管理` 本质上是 Quick BI 的“权限治理底座”。
它不是某一项具体授权动作，而是把 `角色体系、组织级授权开关、集中授权方式、审批流、数据安全边界` 串成一套企业级权限管理框架。官方把这一套框架归纳为：`角色管理及权限、协同授权全局配置、数据协同授权、数据行列权限、集中授权、申请与审批权限、数据安全、权限常见问题`。[权限管理](https://help.aliyun.com/zh/quick-bi/user-guide/permission-management/)

## 这个点解决什么业务问题

- 先解决“企业里谁负责什么权限管理动作”，也就是组织管理员、权限管理员、普通用户、自定义角色的职责边界。[角色管理及权限](https://help.aliyun.com/zh/quick-bi/user-guide/role-permissions) [角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)
- 再解决“组织到底允不允许公开、分享、协同授权、数据集授权、行列权限授权”，也就是组织级总开关问题。[授权设置](https://help.aliyun.com/zh/quick-bi/user-guide/configure-collaborative-authorization)
- 再解决“权限应该按用户发，还是按资源发，能不能批量、能不能同步”，也就是集中授权问题。[集中授权概述](https://help.aliyun.com/zh/quick-bi/user-guide/overview-of-centralized-authorization) [按用户授权概述](https://help.aliyun.com/zh/quick-bi/user-guide/overview-of-granting-permissions-by-user) [从资源赋权概述](https://help.aliyun.com/zh/quick-bi/user-guide/overview-of-granting-permissions-by-resource)
- 最后解决“未授权用户怎么申请权限、谁来审批、审批走哪个渠道”，也就是权限申请闭环问题。[审批设置](https://help.aliyun.com/zh/quick-bi/user-guide/approval-settings)

## 核心功能能力

- `角色化权限框架`：Quick BI 把权限体系建立在 `组织角色 + 空间角色` 双层结构上。组织层预置 `组织管理员、权限管理员、普通用户`，空间层预置 `空间管理员、空间开发者、空间分析师、空间查看者`，并支持自定义角色扩展。[角色管理及权限](https://help.aliyun.com/zh/quick-bi/user-guide/role-permissions) [角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)
- `自定义角色`：专业版支持自定义组织角色和空间角色，能按模块精细定义功能权限，并把用户或用户组批量绑定到指定角色，实现“以岗赋权、多岗多角色”。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)
- `功能权限与资源权限分层`：官方在角色管理页明确把两类权限分开：
  - `功能权限` 决定能不能进入某模块、能不能使用某类能力。
  - `资源权限` 决定对具体资源有没有查看、导出、编辑、使用等权限。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)
- `组织级授权开关`：权限管理员可以在组织级统一控制作品是否允许 `公开、分享、协同授权`，也能控制数据集、数据源、行列权限是否允许授权。如果组织级关掉入口，则所有工作空间都无法再授予对应权限。[授权设置](https://help.aliyun.com/zh/quick-bi/user-guide/configure-collaborative-authorization)
- `默认授权策略`：组织级还能预设新工作空间中 `数据作品查看及导出、数据作品编辑、数据集编辑/使用、数据源编辑/使用` 的默认权限值。例如新报表默认是“仅 Owner 可见”，还是“全部空间成员可查看导出”；新数据集默认是“私密”，还是“有使用权限的人默认可用”。[授权设置](https://help.aliyun.com/zh/quick-bi/user-guide/configure-collaborative-authorization)
- `默认授权时长`：组织级支持把授权有效期统一设成 `1天、3天、1周、1个月` 或自定义时长，这是典型的权限治理约束，不再完全依赖手工记忆收回。[授权设置](https://help.aliyun.com/zh/quick-bi/user-guide/configure-collaborative-authorization)
- `集中授权`：Quick BI 不是只能逐个资源点授权，还提供集中授权模块，从 `资源视角` 和 `用户视角` 两种方式统一管理权限。[集中授权概述](https://help.aliyun.com/zh/quick-bi/user-guide/overview-of-centralized-authorization)
- `按资源集中授权`：权限管理员可以看到组织下所有资源，再按资源类型统一授予查看、导出、编辑或使用权限，还支持把一份权限配置同步到同类型其他资源，支持 `增量` 和 `覆盖` 两种同步模式。[从资源赋权概述](https://help.aliyun.com/zh/quick-bi/user-guide/overview-of-granting-permissions-by-resource)
- `按用户集中授权`：权限管理员可以从用户或用户组视角看到该主体在企业下所有工作空间里的资源权限状态，并把某个用户的权限一键同步到其他人。[按用户授权概述](https://help.aliyun.com/zh/quick-bi/user-guide/overview-of-granting-permissions-by-user)
- `审批流治理`：专业版支持开启权限审批流，决定未授权用户在数据作品页面或列级权限场景下是否能看到“申请权限”入口，并可指定默认审批人。[审批设置](https://help.aliyun.com/zh/quick-bi/user-guide/approval-settings)
- `多渠道审批`：审批可以走 `平台内审批、自定义审批渠道、钉钉、企业微信、飞书`，说明 Quick BI 的权限治理不是平台内孤立动作，而是可接入企业现有协同流转体系。[审批设置](https://help.aliyun.com/zh/quick-bi/user-guide/approval-settings)

## 典型使用场景

- 总部统一管理 BI 权限时，先定义组织角色和空间角色，再通过组织级开关决定哪些授权能力在全公司开放。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management) [授权设置](https://help.aliyun.com/zh/quick-bi/user-guide/configure-collaborative-authorization)
- 多部门共用一个 BI 系统时，权限管理员通过集中授权统一给部门员工分配权限，而不是让每个资源 Owner 分散处理。[集中授权概述](https://help.aliyun.com/zh/quick-bi/user-guide/overview-of-centralized-authorization)
- 企业要求默认最小权限时，可以把新工作空间的作品、数据集、数据源默认授权全部收紧，再按需放开。[授权设置](https://help.aliyun.com/zh/quick-bi/user-guide/configure-collaborative-authorization)
- 企业要求权限申请留痕、审批可追踪、最好还能走钉钉或企业微信时，用审批设置和多渠道审批最合适。[审批设置](https://help.aliyun.com/zh/quick-bi/user-guide/approval-settings)
- 企业需要按岗位定制 BI 能力边界，比如只允许某些人管理企业安全、嵌入分析、数据服务或开放 API 时，自定义角色更适合。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)

## 适用角色

- `组织管理员`：负责权限体系顶层设计，尤其是角色体系、组织级安全与工作空间边界。[角色管理及权限](https://help.aliyun.com/zh/quick-bi/user-guide/role-permissions)
- `权限管理员`：是系统级权限管理的核心执行者，负责集中授权、协同授权配置和权限治理日常运作。[角色管理及权限](https://help.aliyun.com/zh/quick-bi/user-guide/role-permissions) [集中授权概述](https://help.aliyun.com/zh/quick-bi/user-guide/overview-of-centralized-authorization)
- `空间管理员`：更多参与空间内资源权限落实，但不是组织级权限框架的设计者。[角色管理及权限](https://help.aliyun.com/zh/quick-bi/user-guide/role-permissions)
- `资源 Owner`：在协同授权层面会参与资源共享，但不等于系统级权限管理员。

## 关键使用路径

1. 先明确组织和空间的预置角色是否够用；如果不够，再在角色管理里建立自定义组织角色或空间角色。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)
2. 在组织级 `授权设置` 中决定：
   - 是否允许作品公开、分享、协同授权。
   - 是否允许数据集、数据源、行列权限授权。
   - 新工作空间里的默认授权策略是什么。
   - 默认授权时长是多少。[授权设置](https://help.aliyun.com/zh/quick-bi/user-guide/configure-collaborative-authorization)
3. 进入集中授权模块，选择按 `资源视角` 还是按 `用户视角` 做批量赋权。[集中授权概述](https://help.aliyun.com/zh/quick-bi/user-guide/overview-of-centralized-authorization)
4. 如果企业需要申请制访问，再开启审批流，配置入口、默认审批人和审批渠道。[审批设置](https://help.aliyun.com/zh/quick-bi/user-guide/approval-settings)
5. 后续再进入更细的专题能力，比如资源权限、行列权限、申请审批规则逐项落地，这部分就是后面 `权限管理` 专章要展开的内容。[权限管理](https://help.aliyun.com/zh/quick-bi/user-guide/permission-management/)

## 前置条件 / 权限要求 / 版本限制

- `自定义角色`：仅专业版支持，且仅组织管理员可配置；角色相关开放 API 当前只支持系统预置角色，暂不完整支持自定义角色。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)
- `组织级授权设置`：由权限管理员在控制台进行配置；它控制的是组织下所有工作空间的授权入口和默认策略。[授权设置](https://help.aliyun.com/zh/quick-bi/user-guide/configure-collaborative-authorization)
- `协同授权开关`：如果组织级关闭 `可公开 / 可授权`，工作空间里就不会再出现 `作品可公开 / 作品可授权` 相关选项，资源也无法被授予公开、分享和协同授权权限。[授权设置](https://help.aliyun.com/zh/quick-bi/user-guide/configure-collaborative-authorization)
- `审批设置`：仅专业版支持配置审批权限。[审批设置](https://help.aliyun.com/zh/quick-bi/user-guide/approval-settings)
- `多办公软件审批渠道`：如果同一办公软件已接入多个企业，不支持开启对应办公软件审批渠道；反过来，如果对应审批渠道已开启，也不支持再添加多个企业授权。[审批设置](https://help.aliyun.com/zh/quick-bi/user-guide/approval-settings)
- `预置角色功能权限不可改`：组织管理员、权限管理员、普通用户以及预置空间角色的功能权限不能直接改，只能通过自定义角色扩展。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)

## 与上下游点位的关系

- 它的上游是 `账号管理`。因为只有先把用户、用户组、工作空间和角色承载体建立起来，系统级权限框架才有对象可管。[成员管理概述](https://help.aliyun.com/zh/quick-bi/user-guide/overview-of-member-management)
- 它是后面整个 `权限管理` 专章的总入口。后面会分别拆成：
  - `角色权限`：讲角色如何定义和生效。
  - `资源权限`：讲具体资源如何协同授权。
  - `集中授权`：讲批量授权方式。
  - `数据行列权限`：讲细粒度数据安全。
  - `申请与审批权限`：讲权限申请闭环。
- 它和下一点 `系统管理-安全配置` 有边界：这一点偏“权限模型和治理流程”，下一点更偏“安全开关、访问安全、审计和风险控制”。

## 限制与边界

- 这一点不是具体资源授权操作手册。它讲的是权限系统怎么搭，不展开每一种资源对象的逐步授权细节。
- 它也不是账号组织结构本身。用户、用户组、工作空间属于账号管理范畴；这里只把它们当作权限载体。
- 组织级默认授权设置只影响“后续新建的工作空间”，不会回改既有工作空间权限，这是实施时很容易误判的边界。[授权设置](https://help.aliyun.com/zh/quick-bi/user-guide/configure-collaborative-authorization)
- 集中授权虽然高效，但并不是越集中越好。资源 Owner 的协同授权和组织级统一授权需要有职责边界，否则容易出现“系统级改了，业务侧不知道”的治理问题。这一点是根据官方权限结构作出的实施推论。

## 售前 / 方案视角价值解读

- 从售前角度，这一节最重要的信号是：Quick BI 的权限体系不是简单的“给人看报表”，而是完整的企业级权限治理框架，覆盖 `角色、授权入口、默认策略、批量赋权、审批流转`。[权限管理](https://help.aliyun.com/zh/quick-bi/user-guide/permission-management/)
- 从方案设计看，Quick BI 很适合“总部统一治理、各空间按职责分工、资源按需开放”的企业环境，因为它同时有组织级开关、空间级角色、资源级权限和审批机制。[角色管理及权限](https://help.aliyun.com/zh/quick-bi/user-guide/role-permissions) [授权设置](https://help.aliyun.com/zh/quick-bi/user-guide/configure-collaborative-authorization)
- 对客户最有说服力的一点是：Quick BI 不只是支持“谁能看”，还支持“默认怎么开、批量怎么管、申请怎么走、审批怎么接入办公系统”，这更接近真实企业权限运作模式。[集中授权概述](https://help.aliyun.com/zh/quick-bi/user-guide/overview-of-centralized-authorization) [审批设置](https://help.aliyun.com/zh/quick-bi/user-guide/approval-settings)
