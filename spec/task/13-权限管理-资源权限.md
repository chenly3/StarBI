# 第13点：权限管理 - 资源权限

这一点只覆盖 `权限管理-资源权限`。
它和上一点 `角色权限` 的区别很关键：`角色权限` 解决“一个岗位能不能进入某模块、有没有某类功能能力”，`资源权限` 解决“具体这张仪表板、这个数据集、这个数据源，是否对某个人真正开放，以及开放到什么程度”。官方把这类能力统一称为“资源对象权限管理”，资源对象包括 `数据门户、仪表板、电子表格、数据大屏、即席分析、自助取数、数据填报、数据集和数据源`。[资源对象授权的权限类型范围](https://help.aliyun.com/zh/quick-bi/user-guide/overview-of-collaborative-authorization)

## 点位定义
`资源权限` 本质上是 Quick BI 的“对象级授权模型”。
默认情况下，创建者只天然拥有自己创建资源的操作权；当其他人需要查看、导出、编辑或使用这些资源时，就要对具体资源对象做协同授权。[资源对象授权的权限类型范围](https://help.aliyun.com/zh/quick-bi/user-guide/overview-of-collaborative-authorization)

## 这个点解决什么业务问题

- 让企业不用把一个空间里的所有资源都默认开放给所有成员，而是按对象逐一控制开放范围。[资源对象授权的权限类型范围](https://help.aliyun.com/zh/quick-bi/user-guide/overview-of-collaborative-authorization)
- 让“能进模块”和“能用具体资源”分开管理，避免角色权限一放开，所有具体资源都失控。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)
- 支持不同资源采用不同授权粒度，例如报表类资源通常支持 `编辑 / 导出（含查看） / 查看`，而数据类资源更偏 `编辑 / 使用`，数据填报还多出 `管理自身数据 / 管理全部数据 / 提交数据` 这类业务权限。[资源对象授权的权限类型范围](https://help.aliyun.com/zh/quick-bi/user-guide/overview-of-collaborative-authorization)
- 让一份资源可以只开放给指定成员，也可以开放给全部空间成员，或者保持私密状态。[仪表板](https://help.aliyun.com/zh/quick-bi/user-guide/dashboards-1) [电子表格](https://help.aliyun.com/zh/quick-bi/user-guide/workbooks) [授予数据填报权限](https://help.aliyun.com/zh/quick-bi/user-guide/grant-users-the-permissions-on-forms)

## 核心功能能力

- `资源类型覆盖广`：资源权限不是只管报表，官方明确覆盖 9 类对象：`数据门户、仪表板、电子表格、数据大屏、即席分析、自助取数、数据填报、数据集、数据源`。[资源对象授权的权限类型范围](https://help.aliyun.com/zh/quick-bi/user-guide/overview-of-collaborative-authorization)
- `报表类权限模型`：仪表板、电子表格、即席分析、自助取数这类“数据作品”，核心权限是 `编辑 / 导出（含查看） / 查看`。权限之间存在明显包含关系：编辑包含导出和查看，导出包含查看。[资源对象授权的权限类型范围](https://help.aliyun.com/zh/quick-bi/user-guide/overview-of-collaborative-authorization) [仪表板](https://help.aliyun.com/zh/quick-bi/user-guide/dashboards-1) [电子表格](https://help.aliyun.com/zh/quick-bi/user-guide/workbooks)
- `数据大屏权限模型`：大屏更偏展示，不强调导出，所以官方只定义 `编辑 / 查看` 两档，编辑包含查看。[数据大屏](https://help.aliyun.com/zh/quick-bi/user-guide/data-visualization-1)
- `数据集权限模型`：数据集支持 `编辑 / 使用` 两档。
  - 编辑：支持创建、编辑、替换和删除数据集。
  - 使用：支持基于该数据集做数据分析。[授予数据集权限](https://help.aliyun.com/zh/quick-bi/user-guide/grant-users-the-permissions-on-datasets)
- `数据源权限模型`：数据源也支持 `编辑 / 使用` 两档。
  - 编辑：支持创建、编辑、替换和删除数据源。
  - 使用：支持基于该数据源创建数据集。[授予数据源权限](https://help.aliyun.com/zh/quick-bi/user-guide/grant-users-the-permissions-on-data-sources)
- `数据填报权限模型`：数据填报的权限最特殊，包含 `编辑 / 管理全部数据 / 管理自身数据 / 提交数据` 四档，明显更贴近业务填报流程，而不是单纯看/改报表。[授予数据填报权限](https://help.aliyun.com/zh/quick-bi/user-guide/grant-users-the-permissions-on-forms)
- `数据门户双层权限`：数据门户不仅有门户本身的 `编辑` 权限，还有“菜单内容”的 `导出 / 查看` 权限；而且菜单最终生效权限还支持 `菜单 / 内容 / 兼容` 三种判断方式，这是门户资源最特别的地方。[授予数据门户权限](https://help.aliyun.com/zh/quick-bi/user-guide/grant-users-the-permissions-on-bi-portals)
- `私密 / 指定成员 / 全部空间成员`：很多资源在编辑权限上都支持这三种授权属性。
  - `私密`：更偏只有 Owner 或管理角色可操作。
  - `指定成员`：按人定向开放。
  - `全部空间成员`：在具备对应功能权限的前提下，向空间内符合条件的人开放。[仪表板](https://help.aliyun.com/zh/quick-bi/user-guide/dashboards-1) [电子表格](https://help.aliyun.com/zh/quick-bi/user-guide/workbooks) [授予数据填报权限](https://help.aliyun.com/zh/quick-bi/user-guide/grant-users-the-permissions-on-forms)
- `授权主体多元`：至少在数据门户、数据集、数据源、数据填报这些文档中，官方都明确支持 `组织管理员、权限管理员、空间管理员或资源 Owner` 发起授权，这说明资源权限并不完全依赖平台超管，也支持 Owner 直接做协同授权。[授予数据门户权限](https://help.aliyun.com/zh/quick-bi/user-guide/grant-users-the-permissions-on-bi-portals) [授予数据集权限](https://help.aliyun.com/zh/quick-bi/user-guide/grant-users-the-permissions-on-datasets) [授予数据源权限](https://help.aliyun.com/zh/quick-bi/user-guide/grant-users-the-permissions-on-data-sources) [授予数据填报权限](https://help.aliyun.com/zh/quick-bi/user-guide/grant-users-the-permissions-on-forms)
- `跨空间消费支持`：对于仪表板、电子表格、大屏等作品，官方示例明确支持把查看或导出权限授给“与资源不在同一空间”的访问者，这说明资源权限可以突破空间创作边界，把成品分发给其他空间或组织成员消费。[仪表板](https://help.aliyun.com/zh/quick-bi/user-guide/dashboards-1) [电子表格](https://help.aliyun.com/zh/quick-bi/user-guide/workbooks) [数据大屏](https://help.aliyun.com/zh/quick-bi/user-guide/data-visualization-1)
- `数据类资源更严格`：数据源资源最严格，官方只允许“本空间成员中的开发者”被授予编辑或使用权限，分析师、访问者和非本空间成员都不支持；数据集则相对开放，非本空间成员在具备对应功能权限前提下也可以被授予使用权限。[授予数据源权限](https://help.aliyun.com/zh/quick-bi/user-guide/grant-users-the-permissions-on-data-sources) [授予数据集权限](https://help.aliyun.com/zh/quick-bi/user-guide/grant-users-the-permissions-on-datasets)

## 典型使用场景

- 给分析师开放某张仪表板或电子表格的编辑权，让其协同维护正式报表。[仪表板](https://help.aliyun.com/zh/quick-bi/user-guide/dashboards-1) [电子表格](https://help.aliyun.com/zh/quick-bi/user-guide/workbooks)
- 给访问者开放某些报表的查看或导出权限，但不允许其编辑。[仪表板](https://help.aliyun.com/zh/quick-bi/user-guide/dashboards-1)
- 给业务人员开放某个数据集的使用权限，让他们能基于该数据集自助分析，但不允许改数据集定义。[授予数据集权限](https://help.aliyun.com/zh/quick-bi/user-guide/grant-users-the-permissions-on-datasets)
- 给开发者开放某个数据源的使用权限，让其能基于该数据源建数据集；如需改连接配置，再单独开放编辑权限。[授予数据源权限](https://help.aliyun.com/zh/quick-bi/user-guide/grant-users-the-permissions-on-data-sources)
- 在数据填报场景里，把“提交数据”“管理自己提交的数据”“管理全部数据”拆开给不同业务角色，形成业务流程分工。[授予数据填报权限](https://help.aliyun.com/zh/quick-bi/user-guide/grant-users-the-permissions-on-forms)
- 在数据门户里对菜单内容做额外授权，让不同用户看到不同菜单或不同导出能力。[授予数据门户权限](https://help.aliyun.com/zh/quick-bi/user-guide/grant-users-the-permissions-on-bi-portals) [数据门户菜单权限](https://help.aliyun.com/zh/quick-bi/user-guide/bi-portal-menu-permissions)

## 适用角色

- `资源 Owner`：最贴近业务场景，适合对自己负责的资源做协同开放。[授予数据集权限](https://help.aliyun.com/zh/quick-bi/user-guide/grant-users-the-permissions-on-datasets)
- `空间管理员`：适合在空间维度统一管理作品和数据资源授权。[授予数据源权限](https://help.aliyun.com/zh/quick-bi/user-guide/grant-users-the-permissions-on-data-sources)
- `权限管理员`：更适合从全局标准出发管理资源权限策略，并处理批量授权或跨部门访问需求。[资源对象授权的权限类型范围](https://help.aliyun.com/zh/quick-bi/user-guide/overview-of-collaborative-authorization)
- `组织管理员`：更多在组织级兜底或处理跨空间、跨部门场景。

## 关键使用路径

1. 先确认组织级是否已经允许该类资源进行协同授权、公开或相关授权动作。[授权设置](https://help.aliyun.com/zh/quick-bi/user-guide/configure-collaborative-authorization)
2. 再进入具体资源对象的协同授权入口，对某一个资源授权，而不是先做批量同步。[授予数据集权限](https://help.aliyun.com/zh/quick-bi/user-guide/grant-users-the-permissions-on-datasets) [授予数据门户权限](https://help.aliyun.com/zh/quick-bi/user-guide/grant-users-the-permissions-on-bi-portals)
3. 选择权限档位：
   - 报表类：`编辑 / 导出（含查看） / 查看`
   - 数据集/数据源：`编辑 / 使用`
   - 数据填报：`编辑 / 管理全部数据 / 管理自身数据 / 提交数据`
4. 再选择授权属性或授权对象：
   - `私密`
   - `指定成员`
   - `全部空间成员`
   - 某些场景下可选用户组。[仪表板](https://help.aliyun.com/zh/quick-bi/user-guide/dashboards-1) [授予数据填报权限](https://help.aliyun.com/zh/quick-bi/user-guide/grant-users-the-permissions-on-forms) [按用户授权概述](https://help.aliyun.com/zh/quick-bi/user-guide/overview-of-granting-permissions-by-user)
5. 如果这套权限还要批量复用到其他资源，那才进入下一点 `集中授权` 的能力范围。[仪表板](https://help.aliyun.com/zh/quick-bi/user-guide/dashboards-1)

## 前置条件 / 权限要求 / 版本限制

- 资源权限能否生效，不只看资源授权本身，还要看对应用户是否具备相应 `功能权限`。
  - 例如数据集编辑要求其空间角色具备 `数据集新建（编辑）`。
  - 数据集使用要求其空间或组织角色具备 `数据集使用` 功能权限。[授予数据集权限](https://help.aliyun.com/zh/quick-bi/user-guide/grant-users-the-permissions-on-datasets)
- 数据源更严格：即便做资源授权，也只支持本空间开发者获得编辑或使用权限，分析师和访问者不支持。[授予数据源权限](https://help.aliyun.com/zh/quick-bi/user-guide/grant-users-the-permissions-on-data-sources)
- 组织级授权设置如果关闭了 `数据作品可授权、数据集可授权、数据源可授权、行列权限可授权`，资源级入口就无法继续放开。[授权设置](https://help.aliyun.com/zh/quick-bi/user-guide/configure-collaborative-authorization)
- 数据门户菜单权限本身只对已经拥有该门户阅览权限的用户生效；如果对门户本身有编辑权限，菜单权限不会再起限制作用。[数据门户菜单权限](https://help.aliyun.com/zh/quick-bi/user-guide/bi-portal-menu-permissions)
- 用户组并不是所有资源都支持。官方按用户视角文档显示，用户组更常见于 `数据门户 / 数据作品 / 数据填报 / 经营洞察 / 指标洞察`，而 `数据集 / 数据源` 这类数据类资源更偏按用户授权。[按用户授权概述](https://help.aliyun.com/zh/quick-bi/user-guide/overview-of-granting-permissions-by-user)

## 与上下游点位的关系

- 它的上游是 `角色权限`。角色权限先决定你有没有进入某模块、使用某类功能的资格，资源权限再决定你能不能碰到具体对象。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)
- 它和下一点 `集中授权` 的关系是“授权模型”和“授权方式”的关系：
  - 这一点讲“资源有哪些权限类型、谁能被授予什么”。
  - 下一点讲“如何从资源视角或用户视角批量把这些权限发出去”。[资源对象授权的权限类型范围](https://help.aliyun.com/zh/quick-bi/user-guide/overview-of-collaborative-authorization) [集中授权概述](https://help.aliyun.com/zh/quick-bi/user-guide/overview-of-centralized-authorization)
- 它和后面的 `数据行列权限` 有包含关系但不等同。数据集资源权限中的 `编辑 / 使用` 是对象级授权，而行列权限是更细粒度的数据可见范围控制。[资源对象授权的权限类型范围](https://help.aliyun.com/zh/quick-bi/user-guide/overview-of-collaborative-authorization)

## 限制与边界

- `资源权限` 不会替代 `角色权限`。用户即使被授予某资源，也仍受角色功能边界约束。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)
- 不同资源的权限模型差异很大，不能把“报表类三档权限”直接套到数据源、数据集或数据填报上。[资源对象授权的权限类型范围](https://help.aliyun.com/zh/quick-bi/user-guide/overview-of-collaborative-authorization)
- 数据类资源比作品类资源更保守，尤其数据源权限明显更严格，这说明 Quick BI 在底层数据入口上采用了更强收敛策略。[授予数据源权限](https://help.aliyun.com/zh/quick-bi/user-guide/grant-users-the-permissions-on-data-sources)
- 数据门户不是普通报表资源，它还带“菜单内容授权”和“原始内容权限叠加”的特殊逻辑，实施时不能按仪表板思路简单类比。[授予数据门户权限](https://help.aliyun.com/zh/quick-bi/user-guide/grant-users-the-permissions-on-bi-portals)

## 售前 / 方案视角价值解读

- 从售前角度，`资源权限` 这一点最能体现 Quick BI 不是“空间内一刀切开放”的 BI 工具，而是支持对象级精细授权的企业级平台。[资源对象授权的权限类型范围](https://help.aliyun.com/zh/quick-bi/user-guide/overview-of-collaborative-authorization)
- 从方案设计看，Quick BI 的资源权限模型非常适合“总部统一平台、部门各管各资源、跨部门按需共享”的场景，因为它把“能进入模块”和“能用具体对象”彻底拆开了。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)
- 对客户最有说服力的一点是：Quick BI 没有用单一权限模型覆盖所有对象，而是按资源本质分别设计了权限类型。
  - 报表类重协作与消费。
  - 数据类重编辑与使用。
  - 数据填报重业务提交与管理。
  - 数据门户重菜单和内容分层。
这会让权限体系更贴近真实业务，而不是“功能有了但不好用”。
