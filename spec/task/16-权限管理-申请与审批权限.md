# 第16点：权限管理 - 申请与审批权限

这一点只覆盖 `权限管理-申请与审批权限`。
如果前面的 `角色权限 / 资源权限 / 集中授权 / 数据行列权限` 解决的是“管理员如何主动配权限”，那么这一点解决的是“普通用户没有权限时，怎么发起申请；管理员怎么审批；审批结果怎么生效；整个过程怎么留痕”。官方把这条链路明确归纳为 `审批设置` 和 `申请与审批权限` 两部分。[审批设置](https://help.aliyun.com/zh/quick-bi/user-guide/approval-settings) [申请与审批权限](https://help.aliyun.com/zh/quick-bi/user-guide/apply-for-permissions-and-handle-permission-applications)

## 点位定义
`申请与审批权限` 本质上是 Quick BI 的“权限闭环机制”。
它让权限体系不只停留在管理员手工授权，而是变成一个可由业务用户主动发起、由系统流转、由审批人处理、最终自动生效的流程化能力。[申请与审批权限](https://help.aliyun.com/zh/quick-bi/user-guide/apply-for-permissions-and-handle-permission-applications)

## 这个点解决什么业务问题

- 用户访问到没有权限的报表、数据集或敏感字段时，不需要离开系统找管理员线下沟通，而是可以直接在 Quick BI 内发起申请。[审批设置](https://help.aliyun.com/zh/quick-bi/user-guide/approval-settings) [申请与审批权限](https://help.aliyun.com/zh/quick-bi/user-guide/apply-for-permissions-and-handle-permission-applications)
- 管理员不需要被动接收零散 IM 消息或口头申请，而是能在统一入口查看、审批、批量处理待办。[申请与审批权限](https://help.aliyun.com/zh/quick-bi/user-guide/apply-for-permissions-and-handle-permission-applications)
- 企业可以把权限申请接入 `平台内审批 / 自定义审批渠道 / 钉钉 / 企业微信 / 飞书`，让权限申请融入现有协同系统。[审批设置](https://help.aliyun.com/zh/quick-bi/user-guide/approval-settings)
- 对敏感字段脱敏场景，也能开放“查看脱敏字段”的申请入口，使列级权限不再是完全静态的强控，而是可申请、可审批的管理机制。[审批设置](https://help.aliyun.com/zh/quick-bi/user-guide/approval-settings)

## 核心功能能力

- `审批流总开关`：组织可统一开启或关闭审批流。不开启时，权限申请闭环无法成立。[审批设置](https://help.aliyun.com/zh/quick-bi/user-guide/approval-settings)
- `双入口申请`：审批入口分成两类：
  - `数据作品页面申请`
  - `列级权限申请`
  也就是一类面向资源对象，一类面向脱敏字段查看。[审批设置](https://help.aliyun.com/zh/quick-bi/user-guide/approval-settings)
- `数据作品页面申请`：开启后，未授权用户可以在数据作品列表或预览页看到申请入口。官方这里覆盖的数据作品包括 `数据门户、仪表板、电子表格、自助取数、即席分析、数据大屏、数据填报、小Q报告`。[审批设置](https://help.aliyun.com/zh/quick-bi/user-guide/approval-settings)
- `列级权限申请`：开启后，用户在看到脱敏字段时可以发起查看申请，这实际上把“列级脱敏”纳入了审批治理链条。[审批设置](https://help.aliyun.com/zh/quick-bi/user-guide/approval-settings)
- `审批渠道`：Quick BI 支持 `平台内审批、自定义审批渠道、钉钉、企业微信、飞书` 五类渠道。
  其中平台内审批还支持：
  - `允许平台内审批`
  - `仅通知`
  选择“仅通知”时，平台内只能看消息，不能直接审批。[审批设置](https://help.aliyun.com/zh/quick-bi/user-guide/approval-settings) [申请与审批权限](https://help.aliyun.com/zh/quick-bi/user-guide/apply-for-permissions-and-handle-permission-applications)
- `默认审批人机制`：组织级可配置默认审批人，支持多选，官方列出的候选包括：
  - 权限管理员
  - 空间管理员
  - 数据作品所有者
  - 企业安全管理员[审批设置](https://help.aliyun.com/zh/quick-bi/user-guide/approval-settings)
- `申请内容结构化`：申请时可填写 `权限范围、截止日期、申请理由` 等信息。
  以仪表板为例，可申请 `查看及导出` 或 `仅查看`；默认有效期 30 天，可自定义截止时间。[申请与审批权限](https://help.aliyun.com/zh/quick-bi/user-guide/apply-for-permissions-and-handle-permission-applications)
- `申请进度跟踪`：申请人可在 `我的申请` 里查看 `审批中 / 已完结` 状态。[申请与审批权限](https://help.aliyun.com/zh/quick-bi/user-guide/apply-for-permissions-and-handle-permission-applications)
- `审批任务池`：审批人可在 `待我审批 / 我已处理` 中查看待办和历史审批记录，并可单条或批量通过/拒绝。[申请与审批权限](https://help.aliyun.com/zh/quick-bi/user-guide/apply-for-permissions-and-handle-permission-applications)
- `数据集权限申请复用`：对于数据集使用权限，官方明确写了一个很实用的点：申请完成后，后续再使用其他类型的数据分析方式时，无需重复申请该数据集的使用权限。[管理数据集](https://help.aliyun.com/zh/quick-bi/user-guide/manage-dataset)

## 典型使用场景

- 业务人员打开某张仪表板没有查看权限，直接在页面上申请查看或导出权限。[申请与审批权限](https://help.aliyun.com/zh/quick-bi/user-guide/apply-for-permissions-and-handle-permission-applications)
- 分析师需要编辑别人创建的数据集，直接在数据集管理页发起编辑权限申请。[管理数据集](https://help.aliyun.com/zh/quick-bi/user-guide/manage-dataset)
- 用户看到手机号、身份证等字段被脱敏，因业务需要申请查看原始值。[审批设置](https://help.aliyun.com/zh/quick-bi/user-guide/approval-settings)
- 企业希望权限审批直接走钉钉/企业微信/飞书，不想只靠平台内消息时，使用外部审批渠道更合适。[审批设置](https://help.aliyun.com/zh/quick-bi/user-guide/approval-settings)
- 权限 Owner 或管理员日常需要处理大量申请时，可以在待审批池里批量通过或拒绝。[申请与审批权限](https://help.aliyun.com/zh/quick-bi/user-guide/apply-for-permissions-and-handle-permission-applications)

## 适用角色

- `普通用户 / 分析师 / 访问者`：是权限申请的发起方，尤其是在看报表、用数据集、申请敏感字段查看时。[申请与审批权限](https://help.aliyun.com/zh/quick-bi/user-guide/apply-for-permissions-and-handle-permission-applications) [管理数据集](https://help.aliyun.com/zh/quick-bi/user-guide/manage-dataset)
- `资源 Owner`：是最常见的审批人之一，尤其在资源级权限场景下。[申请与审批权限](https://help.aliyun.com/zh/quick-bi/user-guide/apply-for-permissions-and-handle-permission-applications)
- `权限管理员 / 空间管理员 / 企业安全管理员`：在组织启用组织级审批流后，会成为默认审批人候选或实际审批执行者。[审批设置](https://help.aliyun.com/zh/quick-bi/user-guide/approval-settings)

## 关键使用路径

1. 组织先在 `审批设置` 中开启审批流，并决定是否开放 `数据作品页面申请` 和 `列级权限申请`。[审批设置](https://help.aliyun.com/zh/quick-bi/user-guide/approval-settings)
2. 组织再配置默认审批人和审批渠道：平台内、自定义、钉钉、企业微信、飞书。[审批设置](https://help.aliyun.com/zh/quick-bi/user-guide/approval-settings)
3. 用户在资源页、预览页或数据集入口看到申请权限按钮后，填写权限范围、截止日期、申请理由并提交。[申请与审批权限](https://help.aliyun.com/zh/quick-bi/user-guide/apply-for-permissions-and-handle-permission-applications) [管理数据集](https://help.aliyun.com/zh/quick-bi/user-guide/manage-dataset)
4. 审批人在 `待我审批` 中查看申请详情，执行通过或拒绝；若平台仅通知，则转到外部审批渠道处理。[申请与审批权限](https://help.aliyun.com/zh/quick-bi/user-guide/apply-for-permissions-and-handle-permission-applications) [审批设置](https://help.aliyun.com/zh/quick-bi/user-guide/approval-settings)
5. 审批结束后，申请人可在 `我的申请` 中查看结果，权限即时生效或被拒绝。[申请与审批权限](https://help.aliyun.com/zh/quick-bi/user-guide/apply-for-permissions-and-handle-permission-applications)

## 前置条件 / 权限要求 / 版本限制

- `审批设置` 仅专业版支持。[审批设置](https://help.aliyun.com/zh/quick-bi/user-guide/approval-settings)
- 平台内真正可审批的前提是审批渠道里选择了 `允许平台内审批`；如果选的是 `仅通知`，平台内只能看详情，不能操作审批。[审批设置](https://help.aliyun.com/zh/quick-bi/user-guide/approval-settings) [申请与审批权限](https://help.aliyun.com/zh/quick-bi/user-guide/apply-for-permissions-and-handle-permission-applications)
- 如果办公软件接入了多个企业，不支持开启对应办公软件审批渠道；反过来，如果对应审批渠道已开启，也不支持再添加多个企业授权。[审批设置](https://help.aliyun.com/zh/quick-bi/user-guide/approval-settings)
- 数据集权限申请还带有功能权限前提：例如编辑数据集要求申请人本身具备开发权限背景；使用数据集要求其当前角色具备相应分析能力入口。[管理数据集](https://help.aliyun.com/zh/quick-bi/user-guide/manage-dataset)

## 与上下游点位的关系

- 它的上游是 `角色权限 / 资源权限 / 集中授权 / 数据行列权限`。前面几项先定义“权限是什么”，这一点才让“没有权限的人如何得到权限”形成闭环。
- 它和 `集中授权` 的关系是：
  - 集中授权偏管理员主动批量发放。
  - 申请审批偏用户按需触发、管理员被动受理。
- 它和 `数据行列权限` 也直接联动，因为列级脱敏字段可以开放申请入口，不再只是静态隐藏。[审批设置](https://help.aliyun.com/zh/quick-bi/user-guide/approval-settings)

## 限制与边界

- 资源权限申请流程默认是单轮审批，审批完成即刻生效，不是多级审批链。[申请与审批权限](https://help.aliyun.com/zh/quick-bi/user-guide/apply-for-permissions-and-handle-permission-applications)
- 官方文档在“审批人”上存在表述差异：
  - `审批设置` 页写明默认审批人可配置为权限管理员、空间管理员、数据作品所有者、企业安全管理员，并支持多选。[审批设置](https://help.aliyun.com/zh/quick-bi/user-guide/approval-settings)
  - `申请与审批权限` 页又写“审批人为资源 Owner，且不支持配置资源审批人”。[申请与审批权限](https://help.aliyun.com/zh/quick-bi/user-guide/apply-for-permissions-and-handle-permission-applications)
  - `管理数据集` 页则显示申请时可“选择审批人”。[管理数据集](https://help.aliyun.com/zh/quick-bi/user-guide/manage-dataset)
  这说明官方文档在不同入口或不同版本下存在差异。更稳妥的理解是：组织级可以预设审批角色范围，但具体资源入口的审批人表现可能因资源类型和当前组织配置而不同。
- 官方对“可申请对象”也有轻微口径差异：
  - `审批设置` 把数据大屏、小Q报告纳入数据作品页面申请范围。[审批设置](https://help.aliyun.com/zh/quick-bi/user-guide/approval-settings)
  - `申请与审批权限` 页列出的支持对象未显式写入数据大屏和小Q报告。[申请与审批权限](https://help.aliyun.com/zh/quick-bi/user-guide/apply-for-permissions-and-handle-permission-applications)
  因此实施时更适合以实际租户界面为准。

## 售前 / 方案视角价值解读

- 从售前角度，`申请与审批权限` 是 Quick BI 从“可配权限”升级到“可运营权限”的关键一环。很多 BI 产品能做授权，但不一定能把授权申请、审批流、消息通知和外部协同渠道打通。
- 从方案设计看，这一能力尤其适合大型组织和跨部门场景：总部不可能预先为每个人配置所有资源权限，按需申请、按需审批会更符合实际使用过程。
- 对客户最有说服力的一点是：Quick BI 不只是把“权限”做成静态配置，还把它做成“可申请、可审批、可留痕、可接办公系统”的流程化能力。[审批设置](https://help.aliyun.com/zh/quick-bi/user-guide/approval-settings) [申请与审批权限](https://help.aliyun.com/zh/quick-bi/user-guide/apply-for-permissions-and-handle-permission-applications)

## 权限管理章节总结

## 章节定位
Quick BI 的 `权限管理` 不是单一授权页面，而是一整套从 `岗位定义 -> 对象授权 -> 批量运营 -> 数据细粒度控制 -> 申请审批闭环` 的企业级权限治理体系。
如果 `系统管理` 解决的是平台怎么被组织化运行，那么 `权限管理` 解决的就是平台里的“权”如何被结构化设计、精细化发放、持续化治理。

## 章节能力地图

- `角色权限`：定义岗位和功能边界，回答“这个人从职责上能进哪些模块、具备哪些能力”。[角色管理](https://help.aliyun.com/zh/quick-bi/user-guide/role-management)
- `资源权限`：定义具体对象访问边界，回答“这个人到底能不能访问这张报表/这个数据集/这个数据源”。[资源对象授权的权限类型范围](https://help.aliyun.com/zh/quick-bi/user-guide/overview-of-collaborative-authorization)
- `集中授权`：定义批量治理方式，回答“资源很多、人很多时，权限怎么高效运维”。[集中授权概述](https://help.aliyun.com/zh/quick-bi/user-guide/overview-of-centralized-authorization)
- `数据行列权限`：定义数据级安全边界，回答“即使访问同一对象，不同人看到的数据范围能否不同”。[数据行列权限](https://help.aliyun.com/zh/quick-bi/user-guide/row-level-and-column-level-permissions/)
- `申请与审批权限`：定义权限申请闭环，回答“没有权限时，怎么申请，谁审批，审批后如何生效”。[审批设置](https://help.aliyun.com/zh/quick-bi/user-guide/approval-settings) [申请与审批权限](https://help.aliyun.com/zh/quick-bi/user-guide/apply-for-permissions-and-handle-permission-applications)

## 章节内各点关系

1. 先用 `角色权限` 把功能边界按岗位切开。
2. 再用 `资源权限` 把具体对象访问边界收紧。
3. 当资源规模变大时，用 `集中授权` 做批量治理。
4. 当同一对象内部还需要细粒度控制时，用 `数据行列权限` 继续往下收。
5. 当用户仍然没有权限时，再通过 `申请与审批权限` 把治理流程闭环。

也就是说，这一章并不是 5 个分散功能，而是一条完整的权限治理链。

## 对企业落地的价值和边界

- 价值上，这一章说明 Quick BI 的权限体系不是“给报表加个查看按钮”这么简单，而是具备企业级 BI 平台应有的五层治理能力：`岗位、对象、批量、数据、流程`。
- 对售前而言，这一章是证明 Quick BI 能进入大型企业、强合规组织、跨部门共享场景的关键，因为这类客户真正关心的从来不只是“能不能看报表”，而是“谁能看、看到多少、怎么批量发、申请怎么走、是否留痕”。
- 边界上，这一章主要解决“权限如何治理”的问题，不再展开登录认证、租户隔离、审计日志、加密等更广义安全体系；那些属于前面 `系统管理 - 安全配置` 的范围。
