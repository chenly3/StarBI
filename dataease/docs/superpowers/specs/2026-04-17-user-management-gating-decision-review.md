# 用户管理 Gating Decisions Review

## 1. 文档目标

- 本文档承接用户管理 `source inventory`、`详细需求分析`、`阶段 B 差异与决策清单`。
- 目标不是重复需求，而是把进入研发前必须冻结的 5 个决策点压缩成评审用文档。
- 本文档输出的是“推荐基线”，若产品/架构评审有不同结论，需要同步回写后续执行计划。

## 2. 决策总表

| 决策点 | 推荐结论 | 当前状态 |
| --- | --- | --- |
| 前端承载方式 | `remote/plugin-first` | 推荐冻结 |
| 新增用户组织归属 | 保持“切换组织后新增”，表单不新增 `orgId` 字段 | 推荐冻结 |
| 外部用户范围 | 本期不做完整独立流转，但必须保留同入口扩展能力并补原型 | 推荐冻结 |
| `beforeUnmountInfo` 映射 | 进入编码前补一份正式映射表，当前只给推荐草案 | 必须补充确认 |
| 重置密码弹窗 | 当前保持“恢复为初始密码”的确认模式；是否展示默认密码仍待确认 | 必须补充确认 |
| 自定义角色描述字段 | 一期默认不阻塞创建主链路，可后置到二期展示 | 推荐冻结 |

## 3. 分项决策

### 3.1 前端承载方式

**问题**

- 用户管理是否应该直接在当前仓库新建本地 Vue 页面，还是继续沿用现有 `plugin/X-Pack` 远端承载模式。

**候选项**

- 方案 A：`remote/plugin-first`
- 方案 B：本地页面化优先

**推荐结论**

- 选方案 A：`remote/plugin-first`

**证据**

- `core/core-frontend/src/router/establish.ts` 会把 `plugin` 路由统一改写到插件壳组件。
- `core/core-frontend/src/components/plugin/src/index.vue` 运行时从 `window['DEXPack']` 取真实页面组件。
- `core/core-frontend/src/components/plugin/src/ImportXpackTool.ts` 依赖远端 `DEXPackTs.umd.js`。
- 当前仓库没有已落地的本地 `/system/user` 用户管理页面实现。

**影响**

- 下轮实施计划应优先分析“远端 X-Pack 模块接管点”和宿主侧接口补齐点。
- 不能直接把任务切成“新建本地列表页/弹窗页/角色页”这种默认本地化子任务。

**不采纳方案 B 的原因**

- 本地页面化不是“小改造”，而是新的承载方案，需要补前端页面归属、X-Pack 替代策略、社区占位后端能力边界三类设计。

### 3.2 新增用户组织归属

**问题**

- 新增用户是否在表单中显式选择组织，还是维持“先切组织，再在当前组织下新增”。

**候选项**

- 方案 A：保持当前组织隐式上下文
- 方案 B：新增表单显式 `orgId`

**推荐结论**

- 选方案 A：保持当前组织隐式上下文

**证据**

- 官方文档基线是“先选择组织，再在该组织下创建用户”。
- `sdk/api/api-permissions/.../UserCreator.java` 不包含 `orgId`。
- `sdk/api/api-permissions/.../UserApi.java` 提供 `switchOrg`，说明组织上下文通过会话切换建立。
- Calicat 新增用户弹窗也没有显式组织字段。

**影响**

- 新增用户弹窗应在标题区或提示条显式展示“当前组织”，避免系统管理员误建到错误组织。
- 前后端不需要为了一期新增去扩合同。

**不采纳方案 B 的原因**

- 一旦新增显式 `orgId`，就会引入合同扩展、权限校验重审、原型重绘和组织切换语义重构，不适合作为本轮单模块推进前提。

### 3.3 外部用户范围

**问题**

- 用户管理一期是否要把“添加外部用户”完整纳入可用流转。

**候选项**

- 方案 A：一期完整纳入
- 方案 B：一期后置，仅保留入口语义和扩展位

**推荐结论**

- 选方案 B：本期后置完整独立流转，但不得删除同入口扩展能力

**证据**

- 官方文档明确区分“添加组织用户”和“添加外部用户”。
- `RoleApi` 已有 `searchExternalUser`、`mountExternalUser` 合同。
- `core/core-frontend/src/locales/zh-CN.ts` 已有 `out_user_title` 文案。
- 当前 Calicat 仅给出“添加组织用户”双栏选择，没有独立的外部用户弹窗原型。

**影响**

- 信息架构上必须保留“组织用户 + 外部用户”双模式语义，避免后续二次重构入口和文案。
- 实施切片中，本期只落“组织用户加人”主链路；外部用户补原型和最小验收口径后再进入实现。

**不采纳方案 A 的原因**

- 当前前端承载页、原型细节、交互状态和前端包装层都未准备完整，强行纳入会把本轮范围从“用户管理单模块冻结”扩大为“用户来源体系设计”。

### 3.4 `beforeUnmountInfo` 映射

**问题**

- `RoleApi.beforeUnmountInfo` 当前只返回 `Integer`，但前端需要映射到角色移除用户时的 3 类影响提示。

**候选项**

- 方案 A：前端自行猜测数值含义
- 方案 B：进入编码前冻结映射表

**推荐结论**

- 选方案 B：先冻结映射表，再编码

**推荐映射草案**

- `0`：普通移除，仅展示通用确认文案
- `1`：移除后用户失去当前组织全部角色，展示 `clear_in_org`
- `2`：移除后用户失去系统全部角色，展示 `clear_in_system`

**证据**

- 官方文档存在 3 类结果：仅从当前角色移除、从当前组织移除、从系统删除。
- `core/core-frontend/src/locales/zh-CN.ts` 已有 `confirm_unbind_user`、`clear_in_org`、`clear_in_system` 三组文案。
- `RoleApi.beforeUnmountInfo` 未公开枚举，当前协议语义不完整。

**影响**

- 该映射必须写入接口说明或前后端联调约定，否则实现只能靠猜，测试也无法稳定验收。
- 在正式确认前，角色移除用户链路只能停留在分析和联调约定，不能进入最终编码。

**风险说明**

- 上述 `0/1/2` 是基于现有文档和文案的最小闭环推断，不应在未确认后直接固化到代码。

### 3.5 重置密码弹窗

**问题**

- 重置密码时是否展示系统默认密码，以及在哪个阶段展示。

**候选项**

- 方案 A：只做二次确认，不展示默认密码
- 方案 B：确认弹窗展示默认密码并允许复制

**推荐结论**

- 当前不冻结方案 A/B 的最终展示策略。
- 本轮推荐保持“恢复为初始密码”的确认模式，默认不把“展示默认密码”写成既定事实。
- 若产品后续确认需要展示默认密码，再补“确认弹窗展示 + 可复制 + 安全提示”的验收口径。

**证据**

- `UserApi` 与前端 `user.ts` 都已提供 `defaultPwd` 查询接口。
- 官方能力是“重置为默认密码”，不是随机重置后静默下发。
- 现有合同里没有“重置后返回新密码”接口，因此如果未来决定展示，展示时机应在提交前完成。
- 但现有分析基线只把“是否展示默认密码”定义为待确认项，还不能直接冻结为明文展示。

**影响**

- 当前实施切片只能把重置密码定义为“确认恢复为初始密码”。
- 如果后续确认展示默认密码，则需补复制入口、安全提示、可见范围和审计要求。

**当前待确认点**

- 是否在确认弹窗中展示默认密码明文。
- 若展示，是否允许复制、是否需要二次权限校验、是否进入审计日志。

### 3.6 自定义角色描述字段

**问题**

- 创建自定义角色时，`desc` 是否在一期展示并纳入必填/选填口径。

**候选项**

- 方案 A：一期展示，作为可选字段
- 方案 B：一期不展示，后置到二期

**推荐结论**

- 选方案 B：一期不让 `desc` 阻塞创建主链路，后置到二期展示

**证据**

- `RoleCreator` 合同存在 `desc` 字段，但当前 Calicat 原型核心关注点是角色名称和继承类型。
- 详细需求分析已把“描述字段是否一期展示”列为待确认项。

**影响**

- 本期“创建自定义角色”验收口径可先收敛为：名称 + 继承类型 + 创建成功。
- 若后续决定展示 `desc`，不会影响 `typeCode` 主链路，只影响弹窗字段和校验。

**不采纳方案 A 的原因**

- 当前主风险不在描述字段，而在继承关系、权限上限和前端承载方式；把 `desc` 拉入一期只会增加原型和验收分支。

## 4. 推荐冻结顺序

1. 先冻结“前端承载方式”与“新增用户组织归属”，因为它们决定实施切片方向。
2. 再冻结“外部用户范围”与“自定义角色描述字段”，确定一期范围边界。
3. 最后冻结 `beforeUnmountInfo` 映射与“重置密码弹窗”细节，作为实现前联调协议。

## 5. 对下轮执行的直接约束

- 如果继续 `remote/plugin-first`，下轮任务不应直接进入本地 Vue 页面搭建。
- 如果保持隐式组织上下文，新增用户需求文档和原型必须补“当前组织”显式提示。
- 如果外部用户后置，执行计划中必须把“组织用户链路”和“外部用户扩展位”分开验收。
- 如果 `beforeUnmountInfo` 映射未确认，下轮只能停留在分析和接口约定，不进入最终编码。
- 如果重置密码弹窗后续采用默认密码展示模式，需要在交互稿中补复制入口和安全提示。
- 如果 `desc` 后置，本期创建自定义角色验收不得再把“描述字段展示”作为阻塞条件。

## 6. 结论

- 就当前证据而言，用户管理模块已经具备进入“决策冻结评审”的条件，但还不具备直接编码条件。
- 最关键的前置门槛不是页面细节，而是先把承载方式、组织模型、外部用户范围、角色移除提示协议、重置密码展示策略一次定清。
