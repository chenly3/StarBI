# 用户管理一期交付优先 PRD Design

## 1. 文档目标

- 本文档用于沉淀 `组织管理中心 > 用户管理` 模块的一期交付优先 PRD 设计。
- 本文档只覆盖用户管理单模块，不扩展到组织管理中心其他模块。
- 本文档以官方 X-Pack 用户管理能力为参考基线，但以本次一期交付范围为准。
- 本文档是产品需求设计文档，不是实现计划，不进入 `write plan`，也不直接约束编码拆分。

## 2. 设计结论

### 2.1 文档定位

- 文档类型：一期交付优先 PRD
- 需求基线：官方 X-Pack 用户管理能力
- 文档主视角：本次一期交付范围
- 非文档目标：
  - 不做 7 模块总览
  - 不做实现计划
  - 不做接口设计文档

### 2.2 一期交付范围

- 用户基础管理：
  - 用户列表
  - 查询/筛选
  - 新增用户
  - 编辑用户
  - 删除用户
  - 启停用户
  - 重置密码
- 批量导入：
  - 模板下载
  - 批量导入
  - 导入结果反馈
  - 错误报告下载
- 角色视角管理：
  - 角色列表
  - 角色详情成员查看
  - 添加组织用户
  - 从角色移除用户
  - 创建自定义角色

### 2.3 本期非范围

- 添加
- 外部用户搜索与挂载

### 2.4 Calicat 原型输入

- `file_id`: `2044998365299601408`

## 3. PRD 结构设计

### 3.1 正文结构

1. 文档目标
2. 产品范围
3. 用户基础管理
4. 批量导入
5. 角色视角管理
6. 产品规则总表
7. 实现前确认项
8. 非范围项
9. 原型节点映射附录

### 3.2 结构原则

- 正文按能力域展开，不按原型页面顺序平铺。
- 附录负责回映 Calicat 节点，不在正文重复原型说明。
- 正文只写本期交付，不写外部用户后续规划。

## 4. 产品范围设计

### 4.1 范围原则

- 只写本次一期要交付的用户管理能力。
- 以官方 X-Pack 为参考，但不把官方存在而本期不做的能力自动纳入交付。
- 范围必须硬边界化，避免评审时反复把外部用户能力拉回一期。

### 4.2 成功定义

- 系统管理员或组织管理员可以在当前组织上下文下完成：
  - 用户查询
  - 新增/编辑
  - 删除/启停
  - 重置密码
  - 批量导入
  - 角色成员管理
  - 自定义角色创建

只要以上闭环成立，即视为用户管理一期完成。

## 5. 用户基础管理设计

### 5.1 设计目标

- 以“当前组织下的用户管理闭环”为核心，不扩展为全系统统一用户中心。
- 以用户列表页作为主承载页，动作能力围绕列表展开。

### 5.2 章节结构

用户基础管理拆成 7 个功能单元：

1. 用户列表
2. 查询/筛选
3. 新增用户
4. 编辑用户
5. 删除用户
6. 启停用户
7. 重置密码

### 5.3 用户列表设计结论

- 用户列表页应承载：
  - 当前组织下的用户查看
  - 查询/筛选
  - 行级操作
  - 批量操作
  - 新增入口
  - 批量导入口
- 一期字段固定为：
  - 姓名
  - 账号
  - 角色
  - 邮箱
  - 用户来源
  - 启用状态
  - 创建时间
- `所属组织` 不写入一期正文要求字段。

### 5.4 查询/筛选设计结论

- 关键字搜索范围：
  - 姓名
  - 账号
  - 邮箱
- 筛选项：
  - 用户来源
  - 启用状态
  - 角色
- 一期不引入复杂高级检索或跨组织检索。

### 5.5 新增/编辑用户设计结论

- 新增用户发生在当前组织上下文内。
- 新增/编辑表单字段：
  - 账号
  - 姓名
  - 邮箱
  - 手机
  - 角色
  - 是否启用
- 账号创建后不可修改。
- 角色允许多选。
- PRD 中应明确要求弹窗显式提示“当前组织”。
- 编辑用户与新增用户共用表单结构，但账号只读。

### 5.6 删除/启停设计结论

- 删除必须同时支持单个删除与批量删除。
- 删除必须具有确认动作。
- 启停作为列表行级标准操作，不单独拆页面。

### 5.7 重置密码设计结论

- 一期 PRD 中只冻结到“恢复为初始密码”。
- PRD 不把“展示默认密码”写成固定要求。
- 展示策略保留到实现前确认项单独确认。

### 5.8 用户基础管理的写法模板

每个功能单元统一写：

- 功能目标
- 使用角色
- 触发入口
- 页面元素
- 业务规则
- 验收口径

## 6. 批量导入设计

### 6.1 设计目标

- 将批量导入作为一条独立业务流程，而不是用户基础管理中的普通动作。
- 重点保证“模板下载 -> 导入 -> 部分成功 -> 错误报告下载”的主链路闭环。

### 6.2 章节结构

- 流程目标
- 前置条件
- 导入弹窗
- 导入结果反馈
- 核心业务规则
- 验收口径

### 6.3 批量导入设计结论

- 入口：用户列表页 `批量导入`
- 导入弹窗必须包含：
  - 模板下载入口
  - 上传区域
  - 文件格式/大小限制提示
  - 导入按钮
- 一期规则固定为：
  - 支持 Excel 模板导入
  - 文件上限 `10M`
  - 非法文件需要拦截

### 6.4 导入结果设计结论

- 导入结果承接不能只用 toast。
- 结果页或结果弹窗必须包含：
  - 成功数
  - 失败数
  - 错误报告下载
  - 返回用户列表
  - 继续导入

### 6.5 关键业务规则

- 一期必须支持部分成功。
- 合法数据允许先导入成功。
- 非法数据进入错误报告。

## 7. 角色视角管理设计

### 7.1 设计目标

- 从角色视角管理用户成员，而不是展开完整权限体系设计。
- 一期重点是“角色成员管理 + 自定义角色创建”的产品闭环。

### 7.2 章节结构

角色视角管理拆成 5 个功能单元：

1. 角色列表
2. 角色详情成员查看
3. 添加组织用户
4. 从角色移除用户
5. 创建自定义角色

### 7.3 角色列表设计结论

- 页面至少包含：
  - 角色分组/角色列表
  - 角色详情区域
  - 当前角色下用户列表
  - `添加用户` 入口
  - `创建自定义角色` 入口
- 角色分组一期保持：
  - 系统内置角色
  - 自定义角色

### 7.4 添加组织用户设计结论

- 一期只做“添加组织用户”。
- 交互保持双栏选择：
  - 可选用户
  - 已选用户
- 必须支持：
  - 搜索
  - 勾选
  - 清空已选
  - 提交后刷新当前角色成员

### 7.5 从角色移除用户设计结论

- 一期 PRD 中必须保留该能力。
- 移除前必须先做影响预判。
- 提示必须区分 3 类结果：
  - 仅从当前角色移除
  - 从当前组织移除
  - 从系统删除
- 但返回值映射属于实现前确认项，不在正文内展开。

### 7.6 创建自定义角色设计结论

- 一期最小闭环：
  - 角色名称
  - 继承类型/角色类型
  - 创建成功
- `角色描述` 不作为一期阻塞项。
- PRD 中应明确：
  - 自定义角色只能继承组织内置角色能力边界
  - 权限上限不能超过继承角色

### 7.7 角色视角管理的写法模板

每个功能单元统一写：

- 功能目标
- 触发入口
- 页面元素
- 业务规则
- 验收口径

## 8. 产品规则总表设计

### 8.1 设计目标

- 将重复出现的业务规则集中为规则表，避免在各章节中重复散落描述。

### 8.2 写法

规则表字段统一为：

- 规则主题
- 规则内容
- 影响范围

### 8.3 纳入总表的核心规则

- 当前组织上下文
- 账号不可修改
- 角色可多选
- 导入支持部分成功
- 角色移除用户存在 3 类影响
- 自定义角色继承边界

## 9. 实现前确认项设计

### 9.1 设计目标

- 承认现实中的未冻结策略项，但不让正文失焦。

### 9.2 推荐写法

每项统一写：

- 当前结论
- 待确认点
- 不确认会影响什么

### 9.3 当前必须保留的确认项

- `beforeUnmountInfo` 返回值映射
- 重置密码展示策略

## 10. 非范围项设计

### 10.1 设计目标

- 用硬边界明确排除本期不做的能力，防止评审时范围反复拉宽。

### 10.2 写法

- 直接写“本期不包含”
- 直接写“评审与交付不以这些能力为准”
- 不写“后续可以考虑”这类弱表述

### 10.3 非范围清单

- 添加外部用户
- 外部用户搜索与挂载

附录字段统一为：

- 备注

## 12. 文档写作风格

### 12.1 语言原则

- 使用产品语言，不使用实现语言。
- 少写接口、合同、DTO/VO 名称，多写页面行为、业务规则、用户结果。
- 每条需求尽量写成确定句，不写模糊判断句。

### 12.2 功能章节模板

功能章节统一使用以下模板：

- 功能目标
- 使用角色
- 触发入口
- 页面元素
- 业务规则
- 验收口径

### 12.3 验收表达方式

- 验收口径用产品结果句表达，不写成技术测试步骤。
- 例如：
  - 系统管理员可以在当前组织下新增用户并分配角色
  - 管理员可以下载导入错误报告并继续导入
  - 从角色移除用户时，系统会根据影响范围提示不同结果

### 12.4 边界表达方式

- 实现前确认项单独成节，不污染正文。
- 非范围项直接写排除范围，不写后续规划。
- 附录只做原型映射，不重复需求正文。

## 13. 最终结论

- 正文按能力域展开：
  - 用户基础管理
  - 批量导入
  - 角色视角管理
- 非范围项与实现前确认项单独成节。
- Calicat 节点映射放附录。
- 整体风格应保持“产品需求定稿”，而不是“分析稿”或“实现稿”。
