# 系统设置全页面自适应布局统一设计

## 1. 文档目的

本文档用于冻结 StarBI `系统设置` 下所有页面的统一自适应布局规范，覆盖：

- 系统设置壳层
- 用户管理
- 组织管理
- 权限配置
- 问数配置

本轮目标不是重做业务功能，而是统一桌面端页面在不同分辨率下的：

- 自适应布局
- 信息密度
- 字体层级
- 边界感
- 留白控制
- 操作可读性

## 2. 设计边界

### 2.1 适用范围

- 仅面向桌面端
- 最小支持宽度为 `1280px`
- 本轮不覆盖移动端和窄屏专用布局

### 2.2 适配宽度分档

- `1280-1519`：13 寸紧凑桌面档
- `1520-2119`：标准桌面档
- `2120+`：大屏档

### 2.3 页面铺法

- 列表页、树表页、权限矩阵页：采用铺满型布局
- 表单页、弹窗页：采用定宽居中型布局

### 2.4 菜单策略

- 左侧菜单默认展开
- 当菜单展开后正文有效宽度不足时，菜单自动进入悬浮收起态
- 收起后保留可展开入口，不隐藏系统设置导航能力

### 2.5 视觉基调

- 阅读密度：平衡型
- 边界强度：中边界
- 页面头部：平衡型
- 不默认保留无业务价值的描述说明区
- 表格与列表行高：平衡型
- 弹窗与表单：按分档自适应，不采用单一固定尺寸

## 3. 方案比较

### 3.1 方案 A：逐页打补丁

特点：

- 直接针对每个页面单独压间距、调字号、改宽度

优点：

- 启动快
- 适合短期止血

缺点：

- 系统设置下不同页面会形成不同规则
- 后续维护成本高
- 无法建立统一响应式语言

### 3.2 方案 B：统一规范 + 页面类型适配

特点：

- 先定义系统设置统一布局规范
- 再按页面类型落地到各模块

优点：

- 一致性最好
- 后续扩页面时可复用
- 能同时解决壳层、模块页、弹窗页的风格漂移问题

缺点：

- 首轮定义成本略高于逐页修补

### 3.3 方案 C：重做系统设置框架

特点：

- 重写系统设置整体布局框架

优点：

- 理论上一致性最高

缺点：

- 改动面大
- 回归成本高
- 当前阶段风险不必要

### 3.4 最终选择

采用 `方案 B：统一规范 + 页面类型适配`。

原因：

- 本轮目标是统一 `系统设置` 下所有页面，不是修单页
- 当前已经存在多模块并行演进，必须先收口共性规则
- 方案 B 能兼顾落地效率、复用性和回归风险

## 4. 系统设置统一布局规范

### 4.1 壳层结构

- 顶部品牌栏固定高度：`64px`
- 左侧菜单默认宽度：
  - `1280-1519`：`220px`
  - `1520-2119`：`236px`
  - `2120+`：`252px`
- 左侧菜单收起宽度：`64px`
- 当菜单展开后正文有效宽度小于 `980px` 时，自动进入悬浮收起态
- 正文区始终占满剩余宽度，不做整体居中定宽

### 4.2 正文区外边距

- 页面主内容区 padding：
  - `1280-1519`：`10px 12px 12px`
  - `1520-2119`：`12px 16px 16px`
  - `2120+`：`16px 20px 20px`
- 页面内部纵向节奏：
  - 紧邻区块：`12px`
  - 主区块：`16px`
  - 大区块：`20px`
- 禁止以大面积空白作为主要分区手段

### 4.3 页面头部

- 页面标题区总高度控制在 `44-56px`
- 页面标题字号：
  - `1280-1519`：`22px`
  - `1520-2119`：`24px`
  - `2120+`：`26px`
- 页面默认不保留描述说明文案占位
- 标题与操作区优先同排
- 列表页头部优先单行完成
- 表单页可允许两行头部，但不得形成空洞说明区

### 4.4 左侧菜单视觉规则

- 菜单项高度：
  - 展开态：`40-44px`
  - 收起态：`40px`
- 一级菜单字号：`14-15px`
- 菜单图标尺寸：`16px`
- 选中态必须同时具备：
  - 浅底色
  - 文本高亮
  - 左侧高亮条或等效边界增强
- 菜单组间距控制在 `4-6px`

### 4.5 卡片与分区边界

- 卡片圆角：`8-12px`
- 卡片边框：`1px solid #E6EBF2` 级别
- 模块分区优先用：
  - 卡片描边
  - 浅底色
  - hover / active 态
- 不采用重阴影风格
- 页面层级固定为：
  - 页面背景
  - 主卡片
  - 子分区
  - 表格 / 树 / 表单容器

### 4.6 字体层级

- 页面标题：`22 / 24 / 26`
- 区块标题：`15 / 16 / 18`
- 表格表头：`13 / 14 / 14`
- 正文与表格内容：`14 / 15 / 15`
- 次级信息：`12 / 13 / 13`
- 按钮文字：`14 / 15`
- 禁止大量使用 `12px` 作为正文级文字

### 4.7 表格与列表密度

- 表头高度：`44px` 起
- 表格行高：`44-48px`
- 树列表行高：`40-46px`
- 单元格左右 padding：`14-18px`
- 大屏下优先拉宽名称列、主体列、矩阵列
- 禁止让大屏空间主要消耗在无效留白上
- 表格主体区应尽量填满页面主要垂直区域

### 4.8 弹窗与表单规则

- 弹窗宽度分档：
  - `1280-1519`：`680-760px`
  - `1520-2119`：`760-920px`
  - `2120+`：`880-1040px`
- 删除确认弹窗保持小尺寸
- 复杂配置弹窗允许中大尺寸
- 弹窗内部间距：
  - 外层：`20-24px`
  - 区块间：`16-20px`
- 输入框高度：`40-44px`
- `1280-1519` 优先单列或 1.5 列
- `1520+` 再进入标准双列

## 5. 页面类型落地映射

### 5.1 用户管理

- 用户列表页头部压缩为：标题 + 主操作 + 搜索/筛选
- 列表表格铺满正文区
- 筛选区并入工具栏，不独占大块高度
- 角色列表页与用户列表页共用表格密度
- 新增、导入、结果弹窗统一使用分档宽度规则

### 5.2 组织管理

- 页面定位为树形组织列表页
- 顶部仅保留：
  - 标题
  - 添加按钮
  - 搜索框
- 主表格区铺满正文
- 大屏优先放大组织名称列
- 右侧计数、时间、操作列保持紧凑固定宽度
- 添加、编辑、删除弹窗统一跟随弹窗分档

### 5.3 权限配置

- 按用户配置：
  - 左：用户/角色
  - 中：资源类型/资源树
  - 右：权限矩阵
- 按资源配置：
  - 左：资源类型
  - 中：资源列表/树
  - 右：用户/角色 + 权限矩阵
- 权限矩阵始终作为正文主区域
- 行权限/列权限列表页铺满
- 配置弹窗定宽居中
- 规则区优先利用宽度而非堆积上下空白
- 脱敏规则弹窗保持中小定宽，不无限拉伸

### 5.4 问数配置

- 资源管理、主题管理统一按配置型列表页处理
- 顶部仅保留：
  - 模块 tab
  - 当前页必要操作
- 主体区根据当前页面结构采用：
  - 左导航 / 中列表 / 右配置
  - 或上筛选 / 下列表
- 大屏优先放大：
  - 资源列表区
  - 配置主工作区
  - 主题管理主内容区
- 禁止内容仅占页面上半部一小块而下方整屏空白
- 问数配置作为本轮统一规范的优先试点页

## 6. 统一组件映射

以下能力在本轮中应抽象为统一规则，而不是按模块各自定义：

- 左侧系统设置菜单
- 页面标题区
- 列表页工具栏
- 主卡片容器
- 表格密度
- 树列表密度
- 弹窗宽度分档
- 输入框、按钮、标签字号

## 7. 响应式触发规则

### 7.1 `1280-1519`

- 菜单允许自动悬浮收起
- 页面头部压缩
- 操作按钮允许收纳
- 表单优先单列
- 字号使用小一档但不降为难读密度

### 7.2 `1520-2119`

- 使用标准桌面布局
- 菜单默认展开
- 列表、矩阵、配置区正常铺开
- 双列表单和宽配置面板在此档位启用

### 7.3 `2120+`

- 列表页、矩阵页继续扩展正文主区
- 弹窗页和表单页受最大宽度约束
- 额外宽度用于放大主体列，不用于扩大无意义留白

## 8. 实施策略

### 8.1 第一层：系统设置壳层

统一修改：

- 顶部栏高度
- 左侧菜单宽度与响应规则
- 内容区 padding
- 壳层可用宽度计算

目标：

- 先解决菜单层次弱、边界不清、正文可用宽度不足的问题

### 8.2 第二层：页面类型组件

统一修改：

- 页面标题区
- 工具栏
- 表格容器
- 树列表容器
- 配置主卡片
- 弹窗分档尺寸

目标：

- 建立可复用的页面类型规则，而不是继续按页面单修

### 8.3 第三层：模块落地顺序

建议顺序：

1. 问数配置
2. 权限配置
3. 组织管理
4. 用户管理

原因：

- 问数配置当前最典型地暴露“布局稀、空白大、结构轻”的问题
- 权限配置结构复杂，适合验证统一规范是否稳固
- 组织管理、用户管理可在前两者基础上低风险套用

## 9. 验收口径

### 9.1 统一达标标准

以下条件全部满足才视为本轮完成：

- `1280+` 下无明显挤压、重叠、横向错位
- `1440` 下标题、表格、按钮可读性明显提升
- `1728 / 1920` 下内容区能自然铺开，不再只占中上部小块区域
- `2560+` 下列表/矩阵继续扩正文，弹窗/表单不无限拉宽
- 左侧菜单层级、选中态、边界感统一
- 系统设置各子页视觉语言一致

### 9.2 浏览器验证流程

浏览器自动化验证前固定执行：

- 根据前端改动范围执行 `./dataease-web.sh`
- 执行 `agent-browser close --all`
- 再进行页面自动化验证

### 9.3 验证输出

- 每个模块至少提供 `1440` 与 `1920` 两档截图
- 所有截图必须落在当前工程的 `tmp/` 目录下

## 10. 风险与非目标

### 10.1 本轮不做

- 不重做业务功能逻辑
- 不引入移动端专用布局
- 不改变已冻结的原型主体信息架构
- 不把副标题、说明文案当成默认布局组成部分

### 10.2 本轮允许

- 为适配分辨率调整区块比例
- 为提升密度调整字号、间距、描边、列宽、卡片尺寸
- 在不改变业务结构前提下，调整操作区与正文区的相对占比

## 11. 结论

本轮采用“系统设置统一规范 + 页面类型适配 + 模块分层落地”的实施方式。

交付目标不是修单一页面，而是建立 `系统设置` 下所有页面共用的一套桌面端自适应布局语言，使其在 `1280+` 到大屏宽度下都具备稳定的：

- 可读性
- 信息密度
- 模块层次
- 边界识别
- 空间利用效率
