## 1. 文档目标

- 本文档只服务于用户管理模块中的“角色移除用户”阻塞点。
- 它不是实现计划，也不是接口改造方案。
- 目标是把 `beforeUnmountInfo` 在前后端联调前必须确认的协议语义压缩成一页。

## 2. 当前事实

### 2.1 当前已观察到的合同事实

- 接口：`POST /role/beforeUnmountInfo`
- 返回：`Integer`
- 请求体：`UnmountUserRequest`
  - `rid`
  - `uid`

### 2.2 已有前端文案

- `confirm_unbind_user`
- `clear_in_org`
- `clear_in_system`

### 2.3 官方业务基线

- 用户仍有当前组织其他角色：
  - 只从当前角色移除
- 用户在当前组织没有其他角色，但在其他组织仍有角色：
  - 从当前组织移除
- 用户在系统内没有任何其他角色：
  - 从系统删除

## 3. 当前问题

- 后端合同只暴露 `Integer`，没有公开枚举语义。
- 前端知道需要 3 类提示，但不知道每个整数值对应哪一类结果。
- 如果不先确认映射，前端只能猜协议，测试也无法稳定断言。

## 4. 建议确认内容

### 4.1 必须确认的核心问题

1. `beforeUnmountInfo` 是否稳定只返回 3 个离散值。
2. 每个离散值分别对应哪一种业务结果。
3. 是否存在第 4 类结果，例如：
   - 非法角色关系
   - 用户不存在
   - 当前操作者无权限
4. 异常是通过业务码返回，还是直接走接口失败。

### 4.2 推荐最小确认表

| 返回值 | 建议语义 | 前端提示文案 | 后续动作 |
| --- | --- | --- | --- |
| `0` | 仅从当前角色移除 | `confirm_unbind_user` | 用户确认后调用 `unMountUser` |
| `1` | 从当前组织移除 | `clear_in_org` | 用户确认后调用 `unMountUser` |
| `2` | 从系统删除 | `clear_in_system` | 用户确认后调用 `unMountUser` |

| 服务异常如何返回 | 按接口失败处理，不占用 `0/1/2` 分支 |

## 6. 风险说明

- 当前实现以批准计划为权威来源，前后端需继续保持该映射稳定。
- 若后端后续扩展第 4 类业务结果，必须同步更新接口说明与前端分支。
- 运行时异常仍需走接口失败链路，不能与 `0/1/2` 业务影响混用。

## 7. 建议联调口径

### 7.1 前端期望

- `beforeUnmountInfo` 成功返回一个稳定整数值
- 该整数值能唯一映射到一个风险提示分支
- 前端只负责：
  - 调用预判接口
  - 展示对应提示
  - 用户确认后调用 `unMountUser`

### 7.2 后端期望

- 预判接口只回答“移除后影响范围”
- 不把异常与业务分支混在同一返回值空间
- 如果确实需要多于 3 类状态，应同步补枚举说明

### 7.3 产品期望

- 文案必须一一对应业务影响
- 不能把“从当前组织移除”和“从系统删除”混成统一确认

## 9. 若后端不同意整数映射草案

### 9.1 可接受替代方案 A

- 保持 `Integer`
- 但补一份正式枚举说明到接口文档或联调说明单

### 9.2 可接受替代方案 B

- 返回对象而不是裸 `Integer`
- 例如：

```json
{
  "impactType": "CLEAR_IN_ORG"
}
```

### 9.3 当前不建议

- 前端自行猜测数值含义
- 把 3 类影响压成统一删除确认

## 10. 当前前端实现边界

- 远端插件可以固化 `0/1/2` 分支逻辑，但必须严格使用本单据冻结映射。
- 用户确认后统一调用 `unMountUser`，不新增额外业务分支。
- 未经重新确认，不得扩展新的数值语义或合并三类确认文案。

## 11. 本单据希望拿到的最终答复

| 主题 | 希望确认内容 |
| --- | --- |
| 数值空间 | 是否只存在 `0/1/2` 三种值 |
| 数值语义 | 每个值各自对应哪类影响 |
| 异常策略 | 权限不足、关系不存在、服务异常如何返回 |
| 稳定性 | 该映射是否可作为前端实现与测试依据 |

## 12. 结论

- `beforeUnmountInfo` 的 `0/1/2` 映射已冻结，可直接作为 Task 6 编码与验证依据。
- 角色移除用户链路应先调用预判接口，再按本单据文案含义提示用户，最后调用 `unMountUser`。
- 重开该协议的前提是后端合同变化，而不是前端自行调整解释。
