# 智能问数准确率运营闭环设计

## 背景

本设计重新开题，不沿用此前 `Trusted Answer Phase 2` 计划和 QA/review 结论。此前问题在于把验收、trace、replay、snapshot 等工程检查提前当成产品需求，导致需求目标变成“证明系统做过什么”，而不是“让智能问数更准确、更可解释、更可运营”。

这次需求对标 Quick BI 的智能问数产品思路：问数资源、分析主题、语义知识、权限、行权限、列权限不是孤立配置页，而是共同服务于智能问数准确率。用户问数时，运行时要真实消费这些配置；答案不准时，反馈和系统诊断要能回到配置侧，由管理员修正语义知识或生成权限处理建议。

## Quick BI 对标理解

Quick BI 的智能问数思路更接近一套准确率运营体系：

- 数据集/问数资源需要做问数配置，字段质量、推荐问题、知识库、业务逻辑、同义词、正则匹配等都会影响问数准确率。
- 分析主题不是普通文件夹，而是问数时的数据选择边界和业务语义边界。
- 问数权限、分析主题授权、数据集授权和仪表板问数资源管理共同决定用户能问什么。
- 全局配置会影响运行时能力，例如模型配置、语音问答、数据解读、样例数据集展示等，不应只停留在配置页。
- 用户可以查看或展开部分分析过程、取数过程、字段和数据集依据，但普通用户不直接看到底层权限表达式、内部治理证据或管理员诊断。
- 错答调优更偏向管理员维护知识库、问数配置、推荐问题和授权关系，而不是自动修改权限。

转译到当前项目时，一期采用这些产品约束：

- 问数资源必须有学习状态、最后学习时间、所属分析主题和推荐问题配置入口。
- 分析主题要控制资源规模，资源过多时提示拆分主题或设置优先级，避免影响准确率和响应时长。
- 权限至少区分“能不能使用问数功能”“能不能配置问数资源”“能不能问某个主题/资源”。
- 仪表板、数据集、分析主题进入问数时，都要带入各自默认上下文。
- 全局问数开关和模型选择必须进入运行时上下文，避免页面显示已关闭但运行时仍可调用。

## 设计目标

建立一期“智能问数准确率运营闭环”：

```text
管理员配置资源和主题
  ↓
资源变成 AI-ready 可问单元
  ↓
用户问数时运行时消费主题、资源、知识、权限
  ↓
用户看到答案和可展开依据
  ↓
错答、歧义、权限阻断进入修正待办
  ↓
管理员把问题修正为术语、别名、规则、SQL 示例、推荐问题或权限建议
  ↓
后续问数准确率提升
```

一期不追求完整治理平台，也不把 QA、replay、snapshot 作为产品主线。它先解决一个核心问题：配置如何真正提升智能问数准确率，并且错误如何被运营修正。

## 非目标

- 不自动修改用户权限、资源权限、行权限或列权限。
- 不自动启用全局知识。全局影响范围的知识修正必须有更高权限或二次确认。
- 不建设完整 replay、批量评测、版本发布、审计、回滚治理平台。
- 不把前面 QA/review 结论作为需求输入。
- 不重写 SQLBot 底层模型、SQL 生成和执行引擎。
- 不要求一期达到 Quick BI 全量功能，只对齐其核心产品思路。

## 核心对象

### 问数资源

问数资源是 AI-ready 单元，不是简单可问数据列表。

每个问数资源至少包含：

- 资源类型：数据集、数据文件、仪表板资源、组合数据集等。
- 学习状态：未学习、学习中、已学习、学习失败、需优化。
- AI-ready 状态：不可问、可试问、可正式问、需优化。
- 字段质量：字段业务名、字段说明、指标/维度标识、默认聚合方式。
- 推荐问题：用户选择资源后可直接尝试的问题。
- 语义知识：字段别名、业务术语、业务规则、正则匹配、指标口径。
- SQL 示例：典型问题到 SQL 的示例，用于提升生成稳定性。
- 权限状态：资源授权、列权限覆盖、行权限覆盖。
- 质量评分：基于学习状态、知识完整度、反馈、失败率等生成。

资源 AI-ready 状态影响运行时行为：

- `不可问`：不进入问数候选。
- `可试问`：可用于问数，但答案标识为“配置不完整，结果可能不稳定”。
- `可正式问`：正常进入问数候选。
- `需优化`：仍可问，但管理员侧产生优化待办。

AI-ready 状态必须按确定性规则计算，不能靠人工自由填值：

- `不可问`：没有主题绑定、没有可见字段，或资源被管理员显式禁用。
- `可试问`：有基础结构，但学习不完整、知识偏少或反馈缺口较大。
- `可正式问`：字段、知识、推荐问题、权限配置都达到最低标准。
- `需优化`：可以继续问，但最近反馈、失败率或歧义率提示需要管理员处理。

状态迁移规则：

- `未学习` -> `可试问`
- `学习中` -> `可试问` 或 `可正式问`
- `已学习` -> `可正式问`
- `学习失败` -> `需优化`
- `需优化` -> `可试问` / `可正式问` / `不可问`

判定维度：

- 可见字段数量不得为 0。
- 推荐问题数量不得为 0 才能进入 `可正式问`。
- 质量评分低于阈值时进入 `需优化`。
- 管理员可手动下调到 `不可问`，但不能绕过权限裁剪。
- 质量评分只使用学习状态、可见字段完整度、术语覆盖率、SQL 示例覆盖率、最近反馈数、最近失败率、最近歧义率、最近权限阻断率。

### 分析主题

分析主题是智能问数的业务语义边界，不是普通资源分组。

分析主题承担：

- 聚合一组业务相关问数资源。
- 定义主题名称、业务描述、常见分析目标。
- 设置默认资源、推荐资源和资源优先级。
- 组织主题内术语、业务规则、SQL 示例和推荐问题。
- 定义澄清策略：主题不明确、资源多命中、字段歧义时如何反问。
- 定义追问上下文：多轮对话如何继承主题和资源。

运行时不强制用户每次先选主题。更合理的策略是：

- 如果当前页面或会话已有主题，直接使用。
- 如果问题能明显命中一个主题，自动聚焦并提示。
- 如果多个主题都可能命中，先反问用户选择。
- 如果没有主题可用，提示管理员配置或授权。

### 权限、行权限、列权限

权限是运行时强约束，不是学习修正对象。

系统要消费：

- 菜单/功能权限：用户是否能进入智能问数。
- 分析主题权限：用户是否能使用某个主题。
- 问数资源权限：用户是否能问某个资源。
- 列权限：用户能使用哪些字段、指标、维度。
- 行权限：用户能看到哪些数据行或业务范围。

普通用户看到业务化解释：

- “当前基于你有权限的数据生成。”
- “部分字段因权限限制未参与分析。”
- “你当前无权使用该分析主题。”
- “该资源尚未授权，无法用于问数。”

管理员看到诊断详情：

- 哪类权限导致不可问。
- 哪些资源或字段被过滤。
- 行权限是否导致空结果或范围变小。
- 应跳转到哪个权限配置或行列权限配置页面处理。

权限类问题只生成建议，不自动改权限。

### 语义知识

语义知识用于提升模型理解能力。

一期支持的知识类型：

- 字段别名：如 `GMV`、`销售额`、`成交金额` 的映射。
- 业务术语：如 `活跃客户`、`复购率`、`毛利率` 的定义。
- 指标口径：指标计算方式、默认聚合、时间范围约定。
- 业务规则：业务限定、常用过滤条件、特殊逻辑。
- SQL 示例：典型自然语言问题与 SQL 写法。
- 推荐问题：资源或主题下推荐用户提问的问题。

知识作用域：

- 资源级：优先级最高，只影响单个问数资源。
- 主题级：影响主题内多个资源。
- 全局级：影响范围最大，需要更高权限或二次确认。

Quick BI 对标转译：

| Quick BI behavior | StarBI 一期决策 | 理由 | 验收方式 |
|---|---|---|---|
| 数据集/资源问数配置 | 保留 | 这是准确率闭环核心。 | 管理员能看到资源 AI-ready、学习、知识、推荐问题。 |
| 分析主题作为业务边界 | 保留 | 主题是语义边界，不是普通分组。 | 主题可绑定默认资源和优先级。 |
| 多资源命中时澄清 | 保留 | 避免模型盲猜。 | 多主题 fixture 触发反问或自动聚焦。 |
| 问数过程/依据可展开 | 保留，但做安全裁剪 | 用户需要理解结果来源，但不能暴露治理细节。 | 普通用户只见轻量摘要，管理员见更完整诊断。 |
| 全局配置影响运行时 | 保留 | 模型、解读、样例开关必须真生效。 | 关闭后相关动作不可调用。 |
| 数据文件、仪表板等入口问数 | 保留入口语义 | 入口不同，默认上下文不同。 | 各入口自动带入默认主题或资源。 |
| 全量治理/审计/回滚平台 | 延后 | 一期先把准确率闭环做实。 | 留到后续规划。 |

### 修正待办

修正待办是用户反馈和系统诊断的统一处理池。

待办必须记录：

- 用户问题。
- 命中的分析主题和问数资源。
- 实际使用的业务字段、指标、维度。
- 权限过滤摘要。
- 生成结果摘要。
- 用户反馈内容。
- 系统诊断原因。
- 建议修正类型。
- 当前处理状态。
- tenant/workspace。
- source_answer_id。
- submitter。
- owner。
- duplicate_fingerprint。
- severity。
- impact_count。
- created_at。
- updated_at。
- closed_reason。
- linked_knowledge_id。
- linked_relearning_job_id。

待办状态：

- 待处理。
- 处理中。
- 已转知识修正。
- 已转 SQL 示例。
- 已转推荐问题。
- 已转资源重新学习。
- 已转权限建议。
- 已忽略。
- 需要更多信息。
- 已关闭。

状态迁移规则：

- `待处理` -> `处理中`
- `处理中` -> `已转知识修正`
- `处理中` -> `已转 SQL 示例`
- `处理中` -> `已转推荐问题`
- `处理中` -> `已转资源重新学习`
- `处理中` -> `已转权限建议`
- `处理中` -> `已忽略`
- `处理中` -> `需要更多信息`
- `以上状态` -> `已关闭`

重复聚合规则：

- 同一 `duplicate_fingerprint` 的待办只保留一条主记录。
- 后续重复反馈只累加 `impact_count` 和最新时间。
- `已转知识修正` 必须链接已创建知识项，且主记录保持可追溯。

## 运行时消费链路

智能问数运行时必须先组装准确率上下文，再调用 SQLBot 或 DataEase 后端服务。

```text
用户提问
  ↓
识别入口和用户身份
  ↓
构建 AuthorizedAskScope
  ↓
选择或澄清分析主题
  ↓
筛选当前用户可用问数资源
  ↓
检查资源 AI-ready 状态
  ↓
组装语义上下文（仅限 AuthorizedAskScope）
  ↓
生成问数请求
  ↓
SQLBot / DataEase 后端执行
  ↓
返回事实结果 + 可展开依据 + 推荐动作
```

### 入口识别

入口会影响默认上下文：

- 智能问数首页：需要根据用户历史、默认主题、问题内容自动聚焦或反问。
- 分析主题页进入：默认绑定该主题。
- 问数资源页进入：默认绑定该资源和所属主题。
- 仪表板或图表进入：默认绑定当前可问资源和图表上下文。
- 历史对话追问：继承历史会话主题和资源，但要重新按当前权限过滤。

### 主题选择

主题选择规则：

- 已显式选择主题：优先使用。
- 会话已有主题：默认继承。
- 问题只命中一个主题：自动聚焦。
- 问题命中多个主题：反问澄清。
- 没有可用主题：提示无可问业务上下文。

自动聚焦时，普通用户看到轻量提示：

```text
已基于“销售分析主题”理解你的问题。
```

### 资源筛选

资源筛选规则：

- 只选择当前用户有权限的资源。
- 优先选择 AI-ready 状态更高的资源。
- 当多个资源可能匹配且无法判断时反问。
- 资源未学习或质量较低时提示用户，并在管理员侧生成优化待办。
- 资源不满足可问条件时不直接交给 SQLBot 猜测。

### 语义上下文组装

语义上下文包括：

- 全局问数配置：模型选择、数据解读开关、样例数据集开关、追问策略等。
- 当前主题说明。
- 候选资源说明。
- 业务字段、指标、维度。
- 字段别名和业务术语。
- 业务规则和指标口径。
- SQL 示例。
- 推荐问题上下文。
- 用户历史追问上下文。

上下文组装必须按权限裁剪，不允许把用户无权字段、无权资源或隐藏口径放入普通问数请求。

列权限必须做血缘检查：

- 派生指标依赖隐藏字段时，指标必须隐藏或降级。
- SQL 示例依赖隐藏字段时，示例不得进入上下文。
- 字段别名指向隐藏字段时，别名不得进入上下文。
- 可展开依据不得暴露被隐藏字段的存在。

行权限必须作为执行策略生效：

- 行权限表达式不得进入普通用户 prompt。
- 行权限表达式不得出现在普通用户可见依据中。
- 如果 SQLBot 无法安全承接行权限，则由 DataEase 执行侧应用行权限。
- 无法证明行权限生效时，答案必须 fail closed。

### 权限应用

权限应用分三层：

- 资源层：无权资源不进入候选。
- 字段层：无权字段不进入语义上下文和结果展示。
- 数据层：行权限必须在取数前或 SQL 执行侧生效。

如果权限导致无法回答：

- 普通用户看到业务化提示。
- 管理员侧生成权限诊断待办。
- 系统不自动修改权限。

### AuthorizedAskScope

`AuthorizedAskScope` 是运行时的第一道安全边界，必须在任何语义知识、SQL 示例、推荐问题或依据进入请求之前构建完成。

它至少包含：

- 当前用户身份与功能权限。
- 可见的分析主题。
- 可见的问数资源。
- 可见字段、指标、维度。
- 可见的语义知识。
- 可见的 SQL 示例。
- 可见的推荐问题。
- 可见的用户可展开依据。
- 可生成的管理员待办类型。
- 当前行权限执行策略摘要。
- 当前列权限裁剪结果。

未经 `AuthorizedAskScope` 允许的内容，不得进入自然语言提示、SQLBot 请求、答案依据、推荐动作或待办摘要。

### DataEase / SQLBot 边界

DataEase 与 SQLBot 的职责必须分开：

- DataEase 负责身份校验、权限裁剪、主题选择、资源筛选、待办生成和对外暴露的安全依据。
- SQLBot 负责在 DataEase 提供的安全上下文内生成答案、图表、表格和解释。
- SQLBot 不得自行决定可见资源、可见字段或可见权限表达式。
- DataEase 不得把未裁剪的字段、知识或规则原样透传给前端或 SQLBot。

使用的核心对象包括：

- `AuthorizedAskScope`
- `SemanticKnowledgeBundle`
- `SqlBotRequest`
- `ExecutionPolicy`
- `AnswerEvidence`
- `CorrectionTodo`

### 结果返回

结果返回包括：

- 用户问题消息。
- AI 事实回答消息。
- 图表或表格。
- 问数回答文本。
- 可展开依据。
- 推荐追问。
- 数据解读和趋势预测入口。
- 反馈入口。

## 用户可见依据

普通用户采用“默认轻量 + 可展开依据”的方式。

默认轻量依据示例：

```text
基于：销售分析主题 / 门店销售资源
已应用：当前用户权限、行权限、列权限
命中：3 条业务知识、2 条 SQL 示例
数据时间：2026-05-04 10:30
```

展开“分析过程/依据”后可见：

- 系统识别的问题意图。
- 命中的分析主题。
- 使用的问数资源。
- 相关业务字段、指标、维度的业务名。
- 命中的术语、别名、业务规则摘要。
- 是否发生过澄清或自动聚焦。
- 是否基于当前权限生成。
- 取数过程的业务化摘要。

普通用户不可见：

- 底层物理表名。
- 完整 SQL。
- 行权限表达式。
- 内部学习补丁详情。
- 管理员诊断结论。
- 审计信息。

管理员可以在诊断页面看到更完整信息，但应和普通答案页隔离。

可见性矩阵：

| 内容 | 普通用户 | 管理员诊断页 |
|---|---|---|
| 主题名称 | 可见 | 可见 |
| 资源业务名称 | 可见 | 可见 |
| 业务字段名称 | 可见 | 可见 |
| 命中知识摘要 | 可见 | 可见 |
| 命中知识数量 | 不显示精确数量 | 可见 |
| SQL 示例数量 | 不显示精确数量 | 可见 |
| 物理表名 | 不可见 | 仅诊断权限可见 |
| 完整 SQL | 不可见 | 仅诊断权限可见 |
| 行权限表达式 | 不可见 | 仅诊断权限可见 |
| 被过滤字段名称 | 不可见 | 可见 |
| 被过滤资源名称 | 不可见 | 可见 |

## 学习修正闭环

学习修正入口有两类：用户反馈和系统诊断。

### 用户反馈

用户可以对以下内容反馈：

- 事实问数答案。
- 图表类型。
- 表格结果。
- 数据解读。
- 趋势预测。
- 推荐追问。

反馈类型：

- 有用。
- 不准确。
- 字段理解错。
- 业务口径错。
- 图表不合适。
- 数据范围不对。
- 权限导致无法回答。
- 自定义纠错说明。

### 系统诊断

系统自动诊断：

- 多个主题或资源都可能命中，需要澄清。
- 字段别名缺失，导致问题无法匹配。
- 资源未学习。
- 资源 AI-ready 分数低。
- SQL 示例不足。
- 空结果或异常结果。
- 权限、行权限、列权限导致字段或数据不可用。
- 用户连续重试同类问题。
- 数据解读和事实结果可能不一致。

### 管理员处理

管理员可将待办转为：

- 新增字段别名。
- 新增业务术语。
- 新增指标口径。
- 新增业务规则。
- 新增 SQL 示例。
- 新增推荐问题。
- 发起资源重新学习。
- 调整分析主题资源关系建议。
- 权限配置建议。
- 行列权限配置建议。

语义类修正经管理员确认后可以生效。权限类问题只生成建议，并跳转到权限或行列权限配置页面人工处理。

### 防投毒原则

用户反馈不能直接变成生效知识。

一期必须遵守：

- 普通用户只能提交反馈，不能直接修改语义知识。
- 管理员确认后语义修正才生效。
- 全局知识修正需要更高权限或二次确认。
- 同一问题的大量重复反馈需要聚合，不直接生成大量重复待办。
- 恶意或低质量反馈可以被忽略，并保留处理原因。
- 权限类反馈不能自动改权限。

语义修正发布门槛：

- 作用域明确：资源级、主题级或全局级。
- 冲突检查通过：不覆盖已有别名、术语或 SQL 示例。
- 文本安全：别名、术语、规则、推荐问题不包含敏感信息。
- SQL 示例只允许逻辑示例，不允许写死权限绕过或直接泄露物理表。
- 正则或匹配规则必须可解释，长度受限，并有发布前预览。
- 全局级修正需要更高权限或二次确认。
- 草稿态与发布态分离，未发布内容不进入运行时。
- 原始反馈不得直接注入运行时 prompt。

## 页面体验

### 普通用户答案页

问数成功后展示：

- 事实回答文本。
- 图表/表格。
- 轻量依据摘要。
- 可展开分析过程。
- 推荐追问。
- 数据解读入口。
- 趋势预测入口。
- 反馈入口。

错误或降级状态：

- 主题不明确：展示主题选择澄清。
- 资源不明确：展示资源选择澄清。
- 资源未 AI-ready：提示资源配置不完整，并建议切换资源或联系管理员。
- 权限不足：提示当前权限无法使用该主题/资源/字段。
- 空结果：提示可能是权限、筛选范围、时间条件或资源配置问题。
- 全局问数关闭：隐藏或禁用数据解读、趋势预测、继续追问等入口。
- 模型不可用：显示模型不可用提示，并建议重试或联系管理员。
- 学习中：提示资源仍在学习，建议稍后再试。
- 资源已移除：提示资源变更，要求重新选择主题或资源。
- 会话中途权限失效：保留当前答案，但后续追问重新按当前权限过滤。

状态矩阵：

| Trigger | Normal user copy | Admin copy | Primary CTA | Runtime behavior | Todo side effect | Test fixture |
|---|---|---|---|---|---|---|
| No theme match | 请选择一个分析主题 | 主题命中不唯一 | 选择主题 | 暂停执行 | 记录澄清待办 | overlapping themes |
| Resource not AI-ready | 资源配置不完整 | 资源未学习/低分 | 切换资源 | 不直接猜测 | 记录优化待办 | learning failed |
| Permission denied | 当前权限无法使用该项 | 被权限/行权限/列权限拦截 | 联系管理员 | 继续裁剪 | 记录权限诊断待办 | partial access |
| Global query disabled | 当前功能已关闭 | 全局问数开关关闭 | 去配置页 | 禁止调用相关入口 | 记录配置待办 | disabled runtime |
| Model unavailable | 模型暂不可用 | 模型配置失效 | 重试或检查配置 | 阻断调用 | 记录运维待办 | model missing |
| Resource removed | 资源已变更 | 资源已下线/删除 | 重新选择资源 | 终止旧上下文 | 记录变更待办 | removed resource |
| Partial answer | 展示可用部分结果 | 存在权限或数据缺口 | 查看依据 | 返回部分答案 | 记录补全待办 | partial answer |

### 管理员配置页

问数资源配置页应展示：

- 资源列表。
- AI-ready 状态。
- 学习状态。
- 最后学习时间。
- 所属分析主题。
- 字段质量。
- 推荐问题数量。
- 知识数量。
- SQL 示例数量。
- 最近反馈数。
- 最近失败原因。
- 权限覆盖状态。
- 操作：学习、编辑知识、配置推荐问题、配置 SQL 示例、查看待办。

分析主题配置页应展示：

- 主题列表。
- 主题业务说明。
- 关联资源。
- 资源数量和规模风险提示。
- 默认资源。
- 资源优先级。
- 主题级术语/规则/示例。
- 推荐问题。
- 澄清策略。
- 资源质量汇总。
- 最近问数效果。

所有配置页和待办池都必须定义说明性空状态：

- 无资源：说明为什么没有、下一步怎么创建。
- 无主题：说明主题是语义边界而不是普通目录。
- 无待办：说明当前没有需要修正的准确率问题。
- 无权限：说明需要哪类权限而不是直接报空白。

### 修正待办池

修正待办池应支持：

- 按问题类型筛选：字段错、口径错、权限阻断、资源未学习、SQL 示例不足、解读不一致。
- 按主题、资源、用户反馈次数、影响范围排序。
- 查看单条待办上下文。
- 一键转字段别名、术语、业务规则、SQL 示例、推荐问题。
- 跳转资源学习、主题配置、权限配置、行列权限配置。
- 标记已处理、已忽略、需要更多信息。

## 一期范围

一期采用“准确率运营闭环”方案。

一期拆成三个实施里程碑，避免一次性做成大而散的半成品。

### M1: 授权上下文与安全依据

包含：

- `AuthorizedAskScope`。
- DataEase / SQLBot 安全边界。
- 主题自动聚焦和澄清。
- 资源筛选和 AI-ready 状态。
- 权限裁剪后的语义上下文。
- 用户轻量依据与管理员诊断依据。

### M2: 反馈与修正待办池

包含：

- 用户反馈入口。
- 系统诊断入口。
- `CorrectionTodo` schema。
- 重复反馈聚合。
- 权限类问题建议。
- 管理员待办筛选、查看和处理状态。

### M3: 管理员语义修正闭环

包含：

- 待办转字段别名。
- 待办转业务术语。
- 待办转指标口径。
- 待办转业务规则。
- 待办转 SQL 示例。
- 待办转推荐问题。
- 资源重新学习建议。
- 语义修正草稿、校验和发布。

不包含：

- 自动修改权限。
- 自动启用全局知识。
- 完整 replay / 批量评测治理平台。
- 全量审计平台。
- 复杂版本发布和回滚。
- 将 QA review 当成需求输入。

## 成功标准

验收必须使用 fixture，而不是只靠人工观察。

| Fixture | 验收目标 |
|---|---|
| 两个分析主题都可能命中同一问题 | 运行时反问或自动聚焦，并展示安全依据。 |
| 用户有主题权限但缺少某些字段权限 | prompt、SQL 示例、依据、结果字段都不包含无权字段。 |
| 指标依赖隐藏字段 | 指标隐藏或降级，隐藏字段名不出现在用户可见内容。 |
| 用户有行权限过滤条件 | 行权限在执行侧生效，表达式不进入普通用户 prompt 和依据。 |
| 问数资源学习失败 | 资源进入 `需优化`，用户看到可理解提示，管理员侧产生优化待办。 |
| 用户反馈字段理解错误 | 待办按 fingerprint 聚合，管理员可转字段别名或术语修正。 |
| 全局问数关闭 | 数据解读、趋势预测、继续追问等入口不可调用。 |
| 管理员发布语义修正 | 后续相似问题能消费已发布修正，未发布草稿不进入运行时。 |

验收通过后应满足：

- 管理员能看到每个问数资源的 AI-ready 状态和影响准确率的配置缺口。
- 管理员能把分析主题配置为业务语义边界，而不是普通分组。
- 用户问数时，运行时真实消费主题、资源、语义知识和权限配置。
- 普通用户能看到默认轻量依据，并可展开更多业务化依据。
- 普通用户看不到底层 SQL、物理表名、行权限表达式和管理员诊断。
- 错答、字段歧义、业务口径错误、权限阻断能进入修正待办。
- 管理员能把语义类问题修正为知识、SQL 示例、推荐问题或资源学习任务。
- 权限类问题只能生成建议并跳转配置，不会自动修改权限。
- 后续相似问题能消费已确认的语义修正。

## 后续规划

二期可考虑：

- 批量评测和 golden question。
- 配置版本发布和回滚。
- 修正影响范围预览。
- 更完整的治理看板。
- 管理员审计和导出。
- 多主题、多资源命中质量评分。

这些能力不进入一期，避免再次把产品设计变成过大的工程治理计划。
