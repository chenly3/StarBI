# StarBI 智能问数完整路线设计方案

## 1. 总体架构

```
Phase 1 (信任基座)     Phase 2 (分析链路)     Phase 3 (增强闭环)
    ① 推理显性化          ③ 连续分析追问         ⑤ 预测产品化
    ② 主题范围治理        ④ 知识库增强消歧       ⑥ 归因/异常诊断
                                                  ⑦ 问数运营排障
                                                  ⑧ 知识治理增强
```

---

## 2. P0 ② — 分析主题驱动的范围治理

### 2.1 当前状态

- 分析主题存在（CRUD 已完成），但更像"组织入口"而非"强约束对象"
- SQLBot prompt 中"禁止要求额外信息"导致模型在数据源歧义时硬猜
- 会话中当前作用域不显式展示

### 2.2 Quick BI 做法

- 分析主题绑定默认数据集、默认时间、推荐问题
- 对话界面顶部持续展示当前分析对象
- 选择范围作为问数入口（不是聊着聊着再选）

### 2.3 设计方案

**改动点 1：主题强绑定**

分析主题创建/编辑时，必须绑定以下内容才能启用：

| 绑定项 | 说明 | 是否必填 |
|--------|------|---------|
| 默认数据集 | 主题下问数默认使用的数据集 | 是 |
| 默认时间范围 | 如"近30天""本月"等 | 否（有默认值） |
| 推荐问题 | 主题下的推荐提问列表 | 否 |
| 欢迎语 | 进入主题后的欢迎文案 | 否 |
| 仪表盘绑定 | 可选关联的仪表盘 | 否 |

**改动点 2：会话范围显式化**

在聊天界面顶部增加范围状态条：

```
┌──────────────────────────────────────────────────────────┐
│ 🎯 销售分析主题                            [切换主题 ▾]  │
│ 数据集: sales_db (近30天) | 仪表盘: 销售看板            │
└──────────────────────────────────────────────────────────┘
```

- 点击"切换主题"可更换上下文
- 范围变化时问数记录继承变更
- 主题不可用时（如数据集被删除），显示警告状态

**改动点 3：候选对象确认**

当多数据集/多表命中且知识库无法消歧时，在回答前弹出选择：

```
系统检测到多个候选数据源，请确认：
○ 销售数据集 (sales_db) — 包含门店销售数据
○ 订单数据集 (orders_db) — 包含客户订单数据
                     [确认] [取消]
```

Quick BI 的"分析主题"机制减少了这种场景，我们的强绑定也能大幅降低触发频率。

**改动点 4：去歧义 prompt 策略**

移除 prompt 中的：
- "禁止要求额外信息"
- "即使查询条件不完整，也必须生成可行 SQL"

改为：
- "如果数据源/指标存在歧义且无法通过知识库消歧，列出候选供用户选择"

**涉及文件**：
- `dataease/core/core-frontend/src/views/sqlbot-new/` — 聊天界面
- `dataease/core/core-frontend/src/views/visualized/data/query-theme/` — 主题管理
- `SQLBot/backend/templates/template.yaml` — prompt 模板
- `SQLBot/backend/apps/chat/task/llm.py` — 候选确认逻辑

## 3. P0 ③ — 连续分析追问

### 3.1 当前状态

- 有聊天连续性（历史记录，会话恢复）
- 有推荐问题（LLM 自动生成）
- 缺少"分析连续性"——改时间/改指标/改维度不是产品化的操作

### 3.2 Quick BI 做法

- 每轮对话后，LLM 自动生成 "趋势预测" "异常归因" "Top N 品类" 等推荐
- 推荐问题紧贴当前分析结果
- 多模型同步回答增强可信度
- v5.5.1 支持自定义 Prompt 二次解读

### 3.3 设计方案

**改动点 1：分析动作快捷按钮**

每轮回答下方增加 4 个快捷操作：

```
[⏱ 改时间] [📊 改指标] [📍 改维度] [🔍 加筛选]
```

点击后：
- "改时间"：弹出时间选择器，默认选中当前范围
- "改指标"：列出可用的指标切换
- "改维度"：列出可用的维度切换
- "加筛选"：打开筛选条件面板

**改动点 2：上下文继承提示**

追问发送时，在消息上方展示继承的上下文：

```
📎 基于上一轮上下文：时间=本月 | 维度=区域 | 数据源=sales_db
```

**改动点 3：结果绑定推荐**

推荐问题不再用通用问题池，LLM 根据当前结果生成下一步分析建议：

- 当前分析的是"区域销售额" → 推荐"各区域销售额对比""月度趋势""Top 区域"
- 当前分析的是"客户等级分布" → 推荐"各等级消费金额""大客户留存率""客户价值迁移"

**改动点 4：预测/分析触发**

基于 Quick BI 设计，预测和分析完全走**对话内自然语言触发**：

- 用户问 "预测下月销售额会怎么变化" → LLM 自动调用预测模型
- 用户问 "为什么华南区下降了" → LLM 自动调用归因分析
- 每个回答后的推荐问题中自动出现"趋势预测""异常归因"

不设计独立按钮，不做切换模式。

**涉及文件**：
- `SqlbotNewExecutionMetaCard.vue` — 快捷按钮行
- `useSqlbotNewConversation.ts` — 上下文继承、结果绑定推荐逻辑
- `SQLBot/backend/apps/chat/task/llm.py` — 推荐问题生成优化

## 4. P0 ④ — 知识库增强消歧

### 4.1 当前状态

- 术语配置（已迁移）：关键词匹配，依赖用户手动录入
- SQL 示例库（已迁移）：用于 Few-shot 学习
- 缺少语义匹配、历史消歧复用

### 4.2 Quick BI 做法

- 知识库配置（术语、同义词、样例、正则匹配）
- RAG 工作流：检索知识库融入上下文
- 字段质量评估功能
- "一次澄清，长期生效"的知识沉淀

### 4.3 设计方案

**改动点 1：语义匹配增强**

当前术语匹配是精确关键词匹配，需要升级为向量语义搜索：

```
用户问题："大客户的消费情况"
     │
     ▼
语义搜索 → 找到匹配术语 "客户等级 = 'A'"
         → 置信度 0.92
         → 自动注入 prompt
```

- 复用 SQLBot 已有的 pgsql 向量检索能力
- 增加匹配置信度阈值：> 0.8 自动注入，0.5-0.8 提示确认，< 0.5 不注入

**改动点 2：历史消歧复用**

用户在一次对话中澄清的歧义（如选择了具体数据集、明确了指标口径），记录为"消歧历史"：

```
消歧历史表 design:
  - user_id: 用户 ID
  - question_pattern: 问题模式（向量化存储）
  - resolution: 消歧结果（选了什么数据集、什么时间范围等）
  - created_at: 创建时间
  - last_used_at: 最近使用时间
  - use_count: 使用次数
```

后续同类问题自动应用历史消歧结果，不需要再次询问。

**改动点 3：知识优先级规则**

```
知识库命中 > LLM 推断
   │              │
具体业务知识      通用常识
（术语配置明确）   （LLM 自己猜）
```

当知识库有明确映射时，不允许 LLM 覆盖知识库结果。

**改动点 4：知识质量反馈**

管理员可查看每条知识的"有效性"统计：
- 被使用次数
- 被用户更正次数
- 准确率评分
- 支持标记"低质量"触发重新配置提醒

**涉及文件**：
- `SQLBot/backend/apps/terminology/` — 语义搜索升级
- `SQLBot/backend/apps/system/crud/aimodel_manage.py` — 知识优先级逻辑
- `dataease/core/core-frontend/src/views/system/query-config/` — 知识质量展示

## 5. P1 ⑤ — 预测产品化

### 5.1 当前状态

- 后端预测模板存在（`template.yaml` 中的预测部分）
- 前端注释 "暂不支持分析和预测，需要改造前端"
- API 已就绪（`predict_data` 端点）

### 5.2 设计方案

1. **解锁前端入口** — 移除注释，连接预测功能
2. **预测结果展示** — 预测图表 + 预测区间 + 置信度
3. **触发方式** — 自然语言触发（"预测下月销售额"） + 推荐问题自动出现

---

## 6. P1 ⑥ — 归因/异常诊断

### 6.1 设计方案

1. **异常检测** — 自动计算环比/同比变动，超过阈值标记
2. **归因分析** — 异常时 LLM 自动分析维度下钻，找出贡献最大的因素
3. **呈现方式** — "环比下降 12%，主要因华南区下降 28%，其中深圳门店贡献 15%"

---

## 7. P1 ⑦ — 问数运营排障

### 7.1 设计方案

1. **问题反馈闭环** — 每个回答有"有帮助/没帮助"反馈按钮
2. **内部运营看板** — 管理员可见：问数总量、成功率、失败 Top 原因分布、反馈统计
3. **效果回流** — 反馈数据自动回流到术语/SQL 示例优化建议

---

## 8. P1 ⑧ — 知识治理增强

### 8.1 设计方案

1. **批量管理** — 术语批量导入/导出/编辑（已支持上传 Excel，需增强编辑体验）
2. **版本管理** — 术语和 SQL 示例的版本历史，支持回滚
3. **效果统计** — 每条知识的命中率、正确率、用户反馈

---

## 9. 实施顺序

| 阶段 | 能力 | 预估改动量 | 依赖 |
|------|------|----------|------|
| Phase 1 | ① 推理显性化 | SQLBot prompt + SSE 事件 + 前端渲染 | 无 |
| Phase 1 | ② 主题范围治理 | 主题创建页 + 聊天 UI + prompt 修正 | ① |
| Phase 2 | ③ 连续分析追问 | 前端按钮行 + 后端推荐逻辑 | ①② |
| Phase 2 | ④ 知识库消歧 | 语义搜索 + 消歧历史 + 优先级规则 | ② |
| Phase 3 | ⑤⑥⑦⑧ P1 增强 | 解锁现有功能 + 运营看板 | ①②③④ |
