# Query Resource Learning Product Design

> Scope: StarBI `系统设置 -> 问数配置 -> 问数资源管理` 中“学习能力”的产品设计。  
> Goal: 对齐 Quick BI 问数资源学习的核心设计思路，但结合 StarBI 当前的 `DataEase + SQLBot` 融合形态做可实施化收敛。

## 1. Design Intent

本设计只回答一个核心问题：

> “学习”为什么存在，它如何让智能问数的结果更准确、更有效。

这里的“学习”不是训练通用模型，也不是单纯做元数据整理，而是把一个原始数据集加工成一套可供智能问数引擎直接消费的 `问数资产包`，用于提升：

- 资源召回准确率
- 字段理解准确率
- 过滤值匹配准确率
- SQL 生成稳定性
- 问答结果的业务表达质量

因此，学习能力的产品定位不是“多一个学习按钮”，而是：

> 面向智能问数的资源加工、质量治理和持续优化机制。

## 2. Confirmed Scope

本轮已经冻结的学习输入范围如下：

- 数据集本身
- 术语库
- SQL 示例库

本轮不把历史问答反馈作为学习主输入，而是将其作为增强层回流机制单独设计。

换言之，当前主设计采用：

- 主输入：`数据集 + 术语库 + SQL 示例`
- 增强层：`历史问答反馈`

## 3. Design Options

在进入正式设计前，对问题做过三种产品路径评估。

### Option A: 轻量语义画像型

学习只产出字段理解、样本值和推荐问法。

优点：

- 上线快
- 页面较轻
- 风险低

缺点：

- 对问数准确率提升有限
- 对业务术语和 SQL 约束支撑不足

### Option B: 问数资产编排型

学习产出完整问数资产包，包括资源画像、字段语义、值样本、术语映射、SQL 示例锚点、推荐问法、质量评分、任务记录。

优点：

- 最贴近 Quick BI 问数资源学习的设计目的
- 能直接服务问数准确率
- 便于后续接入质量治理与反馈回流

缺点：

- 设计和实现都比轻量方案更重

### Option C: 知识中台型

把学习结果做成统一知识底座，供问数、推荐、解释、运营全部复用。

优点：

- 长期价值高
- 能支撑更多智能化能力

缺点：

- 当前阶段过重
- 容易把需求做散

### Chosen Option

本设计选择 `Option B: 问数资产编排型`。

原因：

- 它直接对齐“让问数更准”的目标
- 不会像轻量方案那样能力不足
- 也不会像知识中台方案那样前期失控

## 4. Product Definition

### 4.1 Query Resource

问数资源是已纳入智能问数范围，并且具备可消费学习结果的数据集对象。

### 4.2 Learning

学习是对数据集、术语和 SQL 示例进行结构化加工，沉淀成一份可供智能问数运行时使用的资产包。

### 4.3 Learning Result

学习结果不是一个布尔状态，也不是一份日志，而是一套结构化问数资产，供运行时做：

- 资源选择
- 字段映射
- 维值匹配
- SQL 约束
- 推荐问法
- 结果解释
- 质量治理

## 5. Core Product Principle

必须始终坚持以下原则：

> 学习不是为了“学完”，而是为了“在用户提问时减少系统猜测”。

智能问数的错误大致来自五类：

- 资源选错
- 字段找错
- 过滤值匹配错
- SQL 生成错
- 结果表达差

学习机制的价值，就是提前沉淀针对这五类错误的资产，让系统在运行时少猜、少漂移、少幻觉。

## 6. Learning Asset Package

每个问数资源学习后，至少应沉淀以下八类资产。

### 6.1 资源基础画像

字段建议：

- 资源 ID
- 资源名称
- 资源别名
- 所属主题
- 创建者 / 责任人
- 数据集描述
- 业务简介
- 更新时间

作用：

- 为资源召回提供基础上下文
- 为页面展示和推荐问法提供展示信息

### 6.2 字段语义画像

字段建议：

- 字段名
- 显示名
- 字段别名
- 原始类型
- 语义类型
- 业务描述
- 是否指标
- 是否维度
- 是否时间字段
- 是否 ID / 编码字段
- 是否敏感字段

作用：

- 帮助问句映射到正确字段
- 为 SQL 生成提供字段使用约束

### 6.3 维值样本资产

字段建议：

- 字段 ID
- 高频值样本
- 典型时间格式
- 常见过滤值别名
- 归一化值

作用：

- 支撑自然语言过滤值匹配
- 解决用户业务用语与真实值不一致的问题

### 6.4 术语映射资产

字段建议：

- 业务术语
- 映射目标类型（资源 / 字段 / 值）
- 映射目标 ID
- 置信度
- 来源（系统学习 / 人工维护）

作用：

- 把业务词映射到系统可执行对象
- 提升语义命中率

### 6.5 SQL 示例锚点资产

字段建议：

- SQL 示例内容
- 对应自然语言问法
- 涉及字段
- 涉及过滤条件
- 适用场景标签

作用：

- 给模型示范高质量 SQL 结构
- 降低幻觉字段、错误聚合和错误过滤

### 6.6 推荐问法资产

字段建议：

- 系统推荐问题
- 人工维护问题
- 对象 / 角色维度推荐问题
- 问题顺序
- 问题状态

作用：

- 降低使用门槛
- 反向定义“资源适合回答什么问题”

### 6.7 学习质量资产

字段建议：

- 总分
- 等级
- 各维度分
- 风险项
- 建议项
- 最近评估时间

作用：

- 决定资源是否适合开放给智能问数
- 决定运行时是否需要降级

### 6.8 学习任务资产

字段建议：

- 任务编号
- 学习版本
- 开始时间
- 结束时间
- 耗时
- 状态
- 失败原因

作用：

- 排障
- 运营
- 重学治理

## 7. Asset Layering

为了避免把所有学习结果堆成一个大 JSON，建议产品上分成四层存储对象：

- 资源主表
- 学习结果表
- 学习质量表
- 学习任务表

这样后续做版本管理、重学、质量回退、失败排查时更清晰。

## 8. Runtime Consumption Flow

学习资产进入智能问数运行时后，建议按七步参与。

### 8.1 问句拆解

系统先拆出：

- 问题对象
- 指标
- 维度
- 过滤条件

依赖资产：

- 资源基础画像
- 字段语义画像
- 术语映射资产

### 8.2 资源召回

系统先判断当前问题最可能归属哪个问数资源，而不是直接生成 SQL。

资源召回依据：

- 资源名称 / 别名命中
- 资源描述相关度
- 术语命中
- 推荐问法相似度
- SQL 示例场景匹配度

依赖资产：

- 资源基础画像
- 推荐问法资产
- 术语映射资产
- SQL 示例锚点资产

### 8.3 字段理解与语义对齐

在资源确定后，系统将业务语言映射到真实字段。

依赖资产：

- 字段语义画像
- 术语映射资产
- 推荐问法资产

### 8.4 过滤值匹配

系统将用户输入的自然语言值映射到真实过滤值。

依赖资产：

- 维值样本资产
- 术语映射资产
- 字段语义画像

### 8.5 SQL 生成约束

系统基于字段角色、SQL 示例和术语映射给模型加护栏，而不是让模型自由生成。

依赖资产：

- SQL 示例锚点资产
- 字段语义画像
- 术语映射资产

### 8.6 结果解释与答案组织

系统用字段别名和业务术语组织最终答案，并给出后续推荐问法。

依赖资产：

- 资源基础画像
- 字段语义画像
- 推荐问法资产

### 8.7 运行时兜底与降级

当链路信心不足时，不应强答，应降级处理。

建议降级策略：

- 资源召回不稳：提示候选资源确认
- 字段理解不稳：提示口径澄清
- 值匹配不稳：提供候选值确认
- SQL 示例不足：降低置信度并限制自由生成
- 质量评分过低：优先推荐问法，不放开自由问数

依赖资产：

- 学习质量资产

## 9. Learning Quality Model

学习质量评分建议拆成六个主维度，总分 100。

| 维度 | 分值 | 含义 |
|---|---:|---|
| 字段完备度 | 20 | 字段名、别名、类型、描述是否齐全 |
| 语义清晰度 | 20 | 时间/维度/指标/ID/敏感字段是否识别准确 |
| 维值可用度 | 15 | 高频值样本是否足够支撑过滤匹配 |
| 术语覆盖度 | 15 | 业务术语是否覆盖主问题表达 |
| SQL 示例支撑度 | 20 | 是否具备高质量 SQL 和问法映射 |
| 问数可用表现 | 10 | 上线后真实成功率、点踩率、纠错率 |

设计逻辑：

- 前五项属于静态质量
- 最后一项属于动态质量

也就是说，资源初次学习就能获得基础分，后续运行表现会继续修正评分。

## 10. Quality Level Governance

评分结果不能只用于展示，必须参与产品行为控制。

| 等级 | 含义 | 运行策略 |
|---|---|---|
| A | 强可用 | 直接开放自由问数 |
| B | 可用 | 开放问数，同时优先推荐标准问法 |
| C | 有限可用 | 允许问数，但增加澄清和确认 |
| D | 不建议开放 | 不走自由问数，只保留治理入口 |

这意味着：

> 学习结果不是“学完即可问”，而是“学完以后判断值不值得问”。

## 11. Re-learning Design

重学触发应拆成三类。

### 11.1 结构类触发

- 字段新增
- 字段删除
- 字段改名
- 字段类型变化
- 数据集 SQL 变化
- 数据源结构变化

### 11.2 知识类触发

- 术语库更新
- SQL 示例库更新
- 推荐问法结构变化

### 11.3 效果类触发

- 问数失败率上升
- 点踩率上升
- 人工纠正率升高
- 值匹配错误集中出现

## 12. Learning State Machine

建议状态为：

- 未开启
- 待学习
- 学习中
- 学习成功
- 学习失败
- 需重新学习
- 已停用

其中：

- `学习失败`：任务未跑成
- `需重新学习`：旧结果仍存在，但已不可信

这是两个不同的产品状态，不能混用。

## 13. Feedback Loop

虽然本轮不把历史问答反馈作为主输入，但必须预留增强层回流机制。

建议采集五类反馈：

- 显式反馈：点赞、点踩
- 纠错反馈：用户改写问题、重新选资源、重新选字段
- 执行反馈：SQL 执行失败、结果为空、字段不存在
- 行为反馈：频繁追问、立即放弃、重复澄清
- 运营反馈：管理员手工修术语、补 SQL 示例、补推荐问题

## 14. Feedback-to-Asset Mapping

反馈不能只记成一条负样本，而要回流到资产层。

| 问题类型 | 回流目标 |
|---|---|
| 资源选错 | 资源基础画像、推荐问法资产 |
| 字段找错 | 字段语义画像、术语映射资产 |
| 过滤值错 | 维值样本资产、术语映射资产 |
| SQL 不稳 | SQL 示例锚点资产 |
| 结果不好理解 | 资源基础画像、推荐问法资产 |

## 15. Re-learning Decision Strategy

不应因为一次点踩就立刻全量重学。

建议策略：

- 单次点踩：记录，不触发
- 同资源同口径连续点踩：进入观察
- 短期集中失败：标记为 `需重新学习`
- 同字段 / 同值映射错误高发：局部重学
- 结构变化：直接全量重学

重学类型建议拆成：

- 全量重学
- 局部重学

局部重学可只重跑：

- 值样本
- 术语映射
- 推荐问法
- SQL 示例锚点

## 16. Resource Management Page Responsibilities

“问数资源管理”页应是学习能力主运营入口，而不是单纯列表页。

### 16.1 Core Fields

- 名称
- 创建者
- 分析主题
- 最后学习时间
- 学习状态
- 质量等级
- 质量分
- 最近失败原因
- 推荐问题数量
- 术语映射数量
- SQL 示例数量
- 最近反馈数量
- 是否需重新学习
- 重学触发原因

### 16.2 Core Actions

- 立即学习
- 重新学习
- 查看任务详情
- 查看质量详情
- 查看反馈摘要
- 快捷提问设置
- 移除

### 16.3 Detail Views

- 预览数据
- 学习任务详情
- 质量详情
- 推荐问题维护
- 反馈摘要

## 17. Boundary of This Design

本设计只定义学习如何服务智能问数，不扩展到：

- 数据门户
- 普通 BI 建模
- 自助取数
- 电子表格
- 数据填报
- 多模型调度总控

这些能力可后续复用学习资产，但不属于当前设计边界。

## 18. Out of Scope

当前不纳入本设计主闭环的事项：

- 把历史问答反馈作为一期学习主输入
- 做统一知识中台
- 做复杂多模型优化器
- 做自动 SQL 示例生成工厂
- 做复杂权限推理引擎

## 19. Success Criteria

如果本设计被正确实现，应该能达到以下产品结果：

- 问数资源不再只是“可查询”，而是“可问、可懂、可推荐”
- 系统在运行时不再从零猜资源和字段
- 同类问题的 SQL 生成稳定性提升
- 资源是否开放给问数有明确质量门槛
- 学习失败、学习过期、学习回流能被运营和治理

## 20. Final Product Statement

与 Quick BI 思路对齐的关键，不是“有学习按钮”，而是以下四点：

- 学习目标是提升智能问数准确率，而不是展示元数据
- 学习结果是一套问数资产，而不是单个任务状态
- 质量评分直接影响资源是否开放给问数
- 反馈回流会驱动重学和资产修正

这四点成立，学习能力的产品方向就不会跑偏。
