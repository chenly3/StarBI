# Quick BI 智能问答对标与 StarBI 智能问数优化设计

## 1. 背景与目标

本次工作的目标不是对 Quick BI 智能问答做逐项复刻，也不是输出一份泛化竞品报告，而是基于 Quick BI 官方帮助文档所体现的核心能力，对 `StarBI 当前整体智能问数能力` 做一次面向产品规划的差距分析，并沉淀为可直接进入需求讨论的优化设计。

本次设计将 `dataease 新问数体验 + SQLBot 底座能力` 合并视为同一个用户可感知产品来评估，不按模块归属拆分判断。

本次设计最终要回答 3 个问题：

1. Quick BI 智能问答的核心能力到底是什么。
2. StarBI 当前哪些能力已经具备，哪些能力偏弱，哪些是关键缺口。
3. 结合现阶段边界，StarBI 应优先补哪些能力，哪些能力应延后。

## 2. 范围与边界

### 2.1 对标范围

本次只纳入 Quick BI 智能问答相关官方帮助文档中稳定出现、且直接构成问数竞争力的核心能力，重点覆盖：

1. 问数入口与数据范围治理
2. 语义治理与轻量知识注入
3. 问答执行与结果可验证性
4. 连续分析与追问
5. 增强分析能力
6. 配置治理与运营闭环

### 2.2 StarBI 现状口径

本次以用户最终会感知到的 `StarBI 智能问数整体能力` 为评估对象，包含：

1. `dataease/core/core-frontend/src/views/sqlbot-new` 对应的新问数入口、主题、范围、结果展示与历史会话体验
2. `dataease/core/core-frontend/src/views/sqlbot` 对应的旧问数结果与解读交互
3. `SQLBot/backend` 与 `SQLBot/frontend` 中已有的 Text-to-SQL、术语、SQL 示例、推荐问题、预测等底层能力

### 2.3 明确边界

本次设计已确认以下边界：

1. 结果可验证、澄清机制、配置治理、排障追踪这类偏底层但强感知能力，纳入需求池。
2. `问数知识库体系` 当前阶段不单独建设，不把其平台化能力列为 P0。
3. 第一阶段优先级主轴为：`先把问得准、答得可信做起来，再补必要连续分析能力`。
4. `问数运营与排障闭环` 放入 P1，不挤进 P0。
5. 结果可验证性按 `中配版` 定义：展示范围依据、问题理解依据、执行摘要，但不在 P0 暴露过重技术链路。

## 4. 分析方法

### 4.1 对标原则

本次不采用“功能点逐项 checklist”方法，而采用 `核心能力分层对标` 方法。

原因如下：

1. 逐项对表容易把 Quick BI 的非核心交互细节也带入需求池，导致范围膨胀。
2. StarBI 当前更需要先补齐核心分析链路，而不是做表面相似度。
3. 用能力分层方式更适合沉淀出 `P0 / P1 / 暂不优先` 的产品判断。

### 4.2 Graphify 视角

参考 Graphify 的知识图谱整理方式，本次将问数能力组织为一个由上游到下游的能力链，而不是零散功能点。

```mermaid
flowchart LR
    A["问数入口与数据范围"] --> B["语义治理与轻量知识注入"]
    B --> C["问答执行"]
    C --> D["结果可验证性"]
    D --> E["连续分析与追问"]
    E --> F["增强分析能力"]
    F --> G["配置治理与运营闭环"]
```

### 4.3 评估状态定义

每个能力点统一用以下状态判断：

1. `已有`：StarBI 已具备稳定可感知能力。
2. `偏弱`：已有雏形，但离产品化体验还有明显距离。
3. `缺失`：当前没有，或虽有零散能力但对用户不可感知。
4. `暂不优先`：存在价值，但不建议进入本阶段核心需求。

## 5. Quick BI 核心能力蓝图

结合官方文档，本次抽象出的 Quick BI 智能问答核心能力如下。

### 5.1 问数入口与数据范围

1. 通过 `分析主题` 组织问数范围，而不是直接把问数建立在裸数据集之上。
2. 让用户先进入稳定的分析上下文，再发起问答。
3. 在问数过程中持续维持当前分析对象、筛选条件与上下文一致性。

### 5.2 语义治理与轻量知识注入

1. 通过术语、同义词、样例、知识配置等方式降低模型误解业务口径的概率。
2. 在问数前完成选表、选字段、业务表达约束，而不是把全部理解压力留给模型实时推断。

### 5.3 问答执行与结果可验证性

1. 不只追求“生成一个结果”，而是让用户理解系统如何理解问题。
2. 当问题存在关键歧义时，必要时先做问题澄清。
3. 让用户能够判断本次结果是否基于正确范围和正确口径。

### 5.4 连续分析与追问

1. 支持围绕当前结果继续分析，而不是每轮都从零开始。
2. 支持对指标、维度、时间、筛选条件进行延续性调整。
3. 推荐下一步问题，而不是只给静态问题池。

### 5.5 增强分析能力

1. 提供数据解读、归因、预测等增强分析深度的能力。
2. 这些能力依附于主分析链路，而不是孤立能力。

### 5.6 配置治理与运营闭环

1. 支持组织级、全局级的问数策略配置。
2. 支持问题沉淀、主题管理、默认行为控制。
3. 支持问题追踪、效果回流和持续优化。

### 6.2 本地证据摘要

#### 6.2.1 问数范围与主题能力

从 `dataease/core/core-frontend/src/locales/zh-CN.ts` 可见，当前 StarBI 已经具备以下范围相关能力骨架：

#### 6.2.2 当前问答与结果能力

从 `dataease/core/core-frontend/src/views/sqlbot-new/useSqlbotNewConversation.ts` 与 `dataease/core/core-frontend/src/views/sqlbot/StarbiResultCard.vue` 可见，当前已经具备：

3. 推荐问题
4. 数据解读
5. 失败态提示
6. 执行详情入口

#### 6.2.3 当前底层策略

从 `SQLBot/backend/templates/template.yaml` 可见，底层已有：

1. 术语注入
2. SQL 示例注入
3. 图表类型推荐
4. 预测模板
5. 查询限制与字段规则

同时也存在一个与 P0 方向冲突的重要现状：

1. 提示词中存在 `禁止要求额外信息`
2. 存在 `即使查询条件不完整，也必须生成可行 SQL`

这表明当前底层策略明显偏向 `优先生成结果`，而不是 `优先消除歧义后再执行`。

从 `SQLBot/frontend/src/i18n/zh-CN.json` 与 `SQLBot/backend/apps/chat/api/chat.py` 可见：

3. 但对话界面仍有 `暂不支持分析和预测，需要改造前端` 的注释

这说明增强分析能力并非完全没有，但离完整产品闭环还有距离。

## 7. 核心短板判断

### 7.1 问数范围治理：已有框架，但未形成真正稳定的前置治理

StarBI 当前不是没有范围能力，而是：

1. 已具备主题、数据集、文件、仪表盘上下文等多个范围入口
2. 但这些能力更像 `可配置入口`，尚未成为保障问数准确率的前置治理机制

核心问题在于：

1. 多候选数据集 / 分析对象命中时，缺少稳定的确认机制
2. 当前会话到底继承了哪些范围，对用户提示不够强
3. 主题虽然存在，但更偏组织入口，而非强分析约束对象

判断：`偏弱`

### 7.2 澄清式问答：当前最大的核心缺口

当前 StarBI 的整体产品逻辑，仍明显偏向：

1. 问题不够完整时尽量直接生成结果
2. 前端虽有“问题还不够具体”的提醒，但没有结构化澄清流程
3. 底层提示词明确鼓励在条件不完整时也生成 SQL

这会导致一个关键问题：

1. 用户拿到的不是“更可信的答案”，而是“一个尽量被拼出来的答案”

这与 Quick BI 的核心能力方向存在明显差距。

判断：`缺失`

### 7.3 结果可验证性：结果存在，但可信度解释层不足

当前 StarBI 已经能返回图表、文本解读和部分失败信息，但还缺少一层对业务用户友好的“解释层”。

缺口主要体现在：

1. 用户不容易快速判断本次用了哪些数据范围
2. 用户不容易确认系统理解出的指标、维度、时间与筛选条件是否正确
3. 用户失败时知道“失败了”，但不够知道是理解错误、范围冲突、执行异常还是权限 / 服务问题
4. 推荐追问还不够紧贴当前结果验证和下一步分析动作

判断：`偏弱`

### 7.4 连续追问：已有聊天连续性，但缺少分析连续性

当前已有历史问数、推荐问题、会话恢复等基础，因此不能定义为完全缺失。

但当前问题在于：

1. 追问没有被产品化为明确的分析动作
2. 用户不清楚本轮追问继承了哪些上下文
3. 推荐追问更像继续聊天，不够像继续分析

判断：`偏弱`

### 7.5 结论排序

按对标 Quick BI 核心能力及用户感知影响排序，本次结论如下：

1. `结构化澄清式问答`：最大缺口
2. `结果可验证性与执行摘要`：最强感知短板
3. `分析主题驱动的范围治理`：骨架已搭，但尚未打透
4. `围绕结果的连续分析追问`：已有雏形，但产品化不足

## 8. P0 需求定义

### 8.1 需求一：分析主题驱动的问数范围治理

#### 目标

先把“用户到底在什么范围里问”管住，使问数准确率建立在稳定分析上下文之上。

#### 需求定义

将 `分析主题` 从配置对象升级为问数主入口，主题负责承载：

1. 默认数据集范围
2. 推荐问题
3. 欢迎语
4. 可选仪表盘绑定
5. 必要时的数据源确认

同时补足两类机制：

1. `候选对象确认`
   当系统识别到多个可能分析对象时，不直接执行，先让用户确认命中范围。
2. `会话范围显式化`
   在会话中持续展示当前主题、数据集、数据源、仪表盘上下文。

#### 解决的核心问题

1. 防止问数从一开始就问偏
2. 防止会话过程中上下文串扰
3. 让主题真正承担问数治理角色

#### 边界

1. 不扩展为知识库体系
2. 不追求跨主题自由联想
3. 只解决问数前置范围稳定性问题

### 8.2 需求二：结构化澄清式问答

#### 目标

将当前“尽量给答案”的策略，升级为“在关键歧义场景下先澄清再执行”。

#### 需求定义

当且仅当缺少关键分析条件时，系统进入结构化澄清流程。P0 只覆盖高价值澄清场景：

1. 时间范围缺失且无法合理默认
2. 指标口径歧义
3. 维度对象歧义
4. 候选数据集 / 分析对象冲突
5. 问题过泛，直接执行只能得到低价值结果

#### 产品规则

1. 能自动补全的，不澄清
2. 不澄清就大概率误答的，必须澄清
3. 澄清问题必须简洁、结构化、可直接选择或补充
4. 澄清完成后继续当前分析链路，不打断会话

#### 解决的核心问题

1. 减少模型乱猜
2. 降低误答成本
3. 提升结果成立概率与用户信任

#### 边界

1. 不做复杂 Agent 编排
2. 不做重型长期记忆
3. 不追求开放式闲聊能力

### 8.3 需求三：结果可验证性与执行摘要

#### 目标

让用户不只看到结果，还能快速判断结果是否建立在正确的范围和正确的理解之上。

#### 需求定义

在结果区补一层 `可验证信息层`，至少包含：

1. `范围依据`
   展示本次使用的主题、数据集、数据源、是否继承仪表盘筛选。
2. `问题理解依据`
   展示系统识别出的时间、指标、维度、过滤条件，以及是否做了默认补全。
3. `执行摘要`
   用业务用户能读懂的方式说明本次分析思路；失败时提供结构化失败类型和建议下一步动作。

#### 解决的核心问题

1. 用户能快速判断这次范围对不对
2. 用户能快速判断系统理解对不对
3. 用户能知道是否值得继续追问

#### 边界

1. P0 不做复杂技术链路全量透出
2. P0 不做重型置信评分体系
3. 对普通用户展示业务解释层，对内部可保留更细日志能力

### 8.4 需求四：围绕结果的连续分析追问

#### 目标

把当前会话从 `聊天连续性` 升级为 `分析连续性`。

#### 需求定义

优先支持 4 类连续分析动作：

1. 改时间范围
2. 改指标
3. 改维度
4. 加 / 减筛选条件

同时补齐两类机制：

1. `继承提示`
   追问时明确提示本轮继承了哪些上下文。
2. `结果绑定推荐追问`
   推荐追问基于当前结果生成下一步常见分析动作，而不是沿用通用问题池。

#### 解决的核心问题

1. 让一轮分析自然延续
2. 降低用户重复表述成本
3. 让推荐追问从“继续聊”转成“继续分析”

#### 边界

1. P0 不做开放式长链分析编排
2. P0 不做跨多个分析对象自由跳转的复杂会话

## 9. 正式分层与优先级

### 9.1 P0 / 第一阶段必须做

#### P0-1：结构化澄清式问答

原因：

1. 这是当前对标 Quick BI 的最大核心缺口
2. 直接决定从“尽量给结果”到“优先避免误答”的产品策略转向

#### P0-1：结果可验证性与执行摘要

原因：

1. 这是当前最强感知短板
2. 直接决定用户是否相信结果、是否愿意继续追问

#### P0-2：分析主题驱动的问数范围治理

原因：

1. 这是问数准确率的前提
2. StarBI 已有骨架，适合在 P0 打透

#### P0-2：围绕结果的连续分析追问

原因：

1. 这是将问数从一次回答变成连续分析过程的关键
2. 优先级略低于前两项主轴，但仍应纳入同一阶段完整链路

### 9.2 P1 / 第二阶段增强

#### 问数运营与排障闭环

包括：

1. 坏 case 回流
2. 执行链路追踪
3. 问题反馈闭环
4. 内部运营看板

#### 预测能力产品化

包括：

1. 预测触发条件
2. 预测结果呈现
3. 预测解释与使用闭环

#### 归因 / 异常诊断增强

包括：

1. 异常解释
2. 原因拆解
3. 对比归因

#### 更系统的知识治理增强

包括：

1. 术语治理增强
2. SQL 示例治理增强
3. 推荐问题治理增强

说明：这里不等于建设完整知识库平台，而是对已有轻量知识注入能力做体系化增强。

### 9.3 暂不建议优先跟进

为避免范围膨胀，本轮明确不优先纳入：

1. 完整问数知识库平台
2. 语音问答
3. 复杂 Agent 式多轮编排
4. 过重的技术链路全量透出
5. 样例数据集、收藏问题等外围增强
6. 跨主题自由跳转的长会话分析

## 10. 交付结构建议

基于本设计，后续正式需求分析文档建议固定输出以下结构：

1. Quick BI 核心能力蓝图
2. StarBI 当前能力映射
3. 关键差距判断
4. P0 需求定义
5. P1 增强项
6. 暂不优先项

每条需求统一输出以下字段：

1. 需求名称
2. 要解决的问题
3. 对标的 Quick BI 核心能力
4. 当前短板
5. 需求范围
6. 预期价值
7. 优先级
8. 建议阶段

## 11. 验收标准

本设计完成后，后续对标分析与需求输出应满足以下标准：

1. 不再以“功能像不像”为主，而以“核心能力是否补齐”为主。
2. 能清楚说明 StarBI 当前哪些能力是 `已有 / 偏弱 / 缺失 / 暂不优先`。
3. 能明确说明为什么 `澄清式问答` 是最大缺口。
4. 能明确说明为什么 `结果可验证性` 是当前最强感知短板。
5. 能把 P0 收敛为 `定范围、能澄清、可验证、可连续追问` 这一条主分析链路。

## 12. 一句话结论

如果用一句话定义本次对标 Quick BI 的产品方向，建议表述为：

`StarBI 第一阶段不追求功能铺开，而是优先补齐“定范围、能澄清、可验证、可连续追问”这 4 个核心能力，先把智能问数从可用提升到可信、可持续分析。`
