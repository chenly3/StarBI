# SQLBot New 数据选择与文件问数收口设计

## 背景

当前 `sqlbot-new` 已经完成了首页态、结果页、历史树、选择数据弹窗、文件上传弹窗、文件详情弹窗等页面结构，并且主数据集问数链路已经接入真实服务。但从最近的页面截图、自动化回归和代码状态来看，`sqlbot-new` 仍然存在一批“静态原型控件未屏蔽、已有真服务未充分利用、部分边缘链路未收口”的问题。

这批问题集中在：

- 数据集选择弹窗
- 数据文件页签
- 文件上传与文件详情链路
- 首页/结果页的主操作聚焦
- 历史与文件链路的最终可上线闭环

本设计只覆盖 `sqlbot-new` 这条新链路，不扩展分析旧 `sqlbot` 整套页面，只在必要时把它当作能力来源。

## 目标

### 业务目标

- 将 `sqlbot-new` 的数据选择与文件问数相关能力从“真假混合状态”收口为“少但真、可上线”的产品状态。
- 屏蔽所有当前没有真实服务支撑的假交互。
- 强化已有真实能力的可见性与可用性。
- 为后续是否新增增强型服务留下清晰边界。

### 交付目标

- 数据集弹窗只保留真实可用的选择能力。
- 数据文件页签只保留真实可用的上传、预览、提问能力。
- 首页与结果页只暴露稳定、可信的主操作路径。
- 文件问数链路完成最终回归闭环。
- 静态原型控件与真实服务能力边界清晰。

## 非目标

本轮不做以下工作：

- 不扩展 `报告 / 搭建 / 搜索` 为真实业务能力。
- 不为 `我的创建 / 常用 / 最近使用 / 排序 / 分页` 立即补后端服务。
- 不一次性修复所有旧历史记录的上下文缺失问题。
- 不扩展 `sqlbot-new` 以外的旧智能问数页。
- 不做旧代码清理与大规模重构。

## 问题定义

### 一、视觉与信息密度问题

- 数据集弹窗在高分辨率下偏小，信息区缩在中间，不像正式工具弹窗。
- 文件页签与上传弹窗也存在同样的密度问题，信息层级不够稳定。
- 首页输入卡和结果页底部输入卡虽然已放大多轮，但仍需围绕真实主操作继续聚焦。

### 二、静态原型残留问题

在当前 `sqlbot-new` 中，仍有一批“看起来能点、实际没有真实交互或没有真实服务”的控件：

- 数据集弹窗：
  - `我的创建`
  - `常用`
  - `最近使用`
  - 搜索框
  - 排序按钮
  - 分页按钮
  - 卡片右上角 `···`
  - 静态 `预览`
- 数据文件页签：
  - 搜索框
  - `全部 / Excel / CSV / 最近上传`
  - 排序按钮
  - 卡片右上角 `···`
  - 删除图标

这些控件如果继续暴露，会持续制造“产品不可信”的感知。

### 三、已有真服务未接满问题

`sqlbot-new` 中有一批能力并不是后端缺失，而是前端只接了一半，或者已经能用但没有被收口成正式产品入口：

- 数据集真实列表
- 数据集真实选择
- 文件真实上传
- 文件真实解析
- 文件真实预览
- 文件真实问数
- 历史真实读取
- 历史单条删除
- 推荐问题
- 数据解读
- 追加追问

### 四、真正需要服务增强的问题

如果未来坚持保留以下高级筛选能力，则当前服务返回字段和查询能力不够：

- `我的创建`
- `常用`
- `最近使用`
- 真排序
- 真分页

这些能力不应该在本轮作为上线阻塞，而应该单独进入增强阶段。

## 能力分层设计

### 第一层：立即保留的真实能力

以下能力保留，并作为当前主操作核心：

- 数据集真实选择
- 文件真实上传
- 文件真实预览
- 文件真实问数
- 历史查看
- 历史单条删除
- 清空确认弹窗
- 推荐问题
- 数据解读
- 追加追问

这些能力对应的页面入口和视觉层级应继续保留，并在后续收口中作为重点强化对象。

### 第二层：立即屏蔽的静态假能力

以下控件在本轮应直接屏蔽，不再以正式功能姿态暴露：

- 数据集弹窗中的：
  - `我的创建`
  - `常用`
  - `最近使用`
  - 排序
  - 分页
  - 搜索框
  - 卡片更多菜单
  - 静态预览入口
- 数据文件页签中的：
  - 搜索框
  - 筛选 chip
  - 排序
  - 删除图标
  - 更多图标

收口原则：

- 可见即可信
- 可点即有效
- 没接服务的一级交互，先不露出

### 第三层：用前端本地能力补齐的低成本功能

以下能力不急着新增后端服务，可以先在前端基于当前已加载数据做本地能力：

- 数据集搜索
- 文件搜索

设计原则：

- 只做最小能力，不做复杂语义搜索
- 仅对当前已加载列表做本地过滤
- 不同时保留伪排序、伪分页等易误导控件

### 第四层：后端增强阶段

如果后续确认要保留高级筛选能力，建议采用“一个统一增强服务”而不是零散补多个接口。

建议增强能力：

- `keyword`
- `mine`
- `common`
- `recent`
- `sort`
- `pageNum`
- `pageSize`

推荐策略：

- 优先做一个统一的数据集/文件列表查询增强服务
- 避免后续前端为每个筛选条件单独绑定一个零散接口

## 服务现状评估

### 当前无需新增服务即可支撑的能力

- 数据集列表拉取
- 数据集选择
- 数据集 datasource 解析
- 文件上传
- 文件解析
- 文件详情预览
- 文件问数
- 历史读取
- 历史单条删除
- 数据解读
- 追加追问
- 推荐问题

### 当前建议新增服务数量

短期建议新增服务数量：`0`

本轮重点是：

- 屏蔽静态假功能
- 收口已有真能力
- 完成文件链路和历史链路回归

如果进入增强阶段，再考虑新增：

- `1` 个统一增强型列表查询服务

## 实施策略

### 阶段 1：立即整改

- 放大数据集弹窗尺寸与密度
- 屏蔽所有静态假控件
- 保留真实入口
- 修正错误态文案

### 阶段 2：补齐已有真链路

- 文件删除优先复用现有 datasource delete 能力
- 数据集搜索与文件搜索优先做前端本地过滤
- 历史恢复继续只保证“新生成记录”完整可恢复

### 阶段 3：增强阶段

- 若确认保留高级筛选，再补统一增强服务

## 验收标准

### 页面级

#### 首页态

- 只保留真实可用入口
- 用户能明确看到 `选择数据` 与 `直接提问` 两条主路径
- 不存在点击无反应的筛选/管理型控件

#### 结果页

- 成功卡与失败卡均使用产品语言
- 推荐问题、输入卡、结果卡构成稳定主链路
- `数据解读 / 追加追问 / 更多` 稳定可用

#### 选择数据弹窗

- 数据集弹窗只保留真实可用操作
- 不再显示静态分页、静态排序、静态筛选
- 在高分辨率下清晰可读

#### 数据文件页签

- 文件列表、上传、预览、提问都能真实闭环
- 不再暴露未接服务的管理型控件

### 功能级

#### 数据集问数

必须通过：

- 选择数据集
- 发起问数
- 生成结果卡
- 推荐问题出现并可点击
- 数据解读请求发出
- 追加追问回填输入框
- 历史生成

#### 文件问数

必须通过：

- 上传文件
- 预览文件
- 保存 datasource
- 发起问数
- 生成结果卡

#### 历史管理

必须通过：

- 历史列表可见
- 单条删除真实生效
- 清空确认弹窗正常出现
- 新生成历史记录可恢复

### 静态/动态边界

最终必须满足：

- 所有一级操作都具备真实结果
- 没接服务的控件不再保留为正式功能入口
- 页面中不存在“看起来像功能但实际只是原型”的交互

## 风险与控制

### 风险 1：前端只屏蔽假功能，但真功能链路回归不足

控制：

- 所有保留入口必须跑浏览器回归

### 风险 2：历史与文件链路边缘状态遗漏

控制：

- 单独把文件链路和历史链路列为最终验收矩阵

### 风险 3：后端增强范围失控

控制：

- 本轮不新增服务
- 增强需求统一进入下一阶段设计

## 最终结论

`sqlbot-new` 当前最合理的收口方式不是“把所有原型控件做真”，而是：

1. 先屏蔽所有静态假功能
2. 把已有真链路全部收稳
3. 再决定哪些高级筛选能力值得补服务

这个方案能最短路径把 `sqlbot-new` 推到一个“可上线、可验收、可信”的状态。
