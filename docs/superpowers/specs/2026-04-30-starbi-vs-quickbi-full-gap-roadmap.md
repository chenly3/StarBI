## 2. 核心差距详析（基于用户可感知的产品功能）

### 2.1 🔴 最核心缺失：澄清式问答

**当前状态**：用户提模糊问题 → 系统尽力拼出结果 → 用户不确定是否可信

**Quick BI 做法**：缺少时间/指标/维度等关键条件时，先给选项让用户确认，再执行

**当前 StarBI**：
- prompt 中有 "禁止要求额外信息"
- 有 "即使查询条件不完整，也必须生成可行 SQL"
- 这导致模型在歧义场景下"猜"而非"澄清"

**影响**：这是对标 Quick BI 的最大结构性落差，直接决定了从"可用"到"可信"的跨越。

### 2.2 🟠 最影响信任：结果可验证性

**当前状态**：用户看到结果，但不知道系统怎么理解的

**需要补的**：
- 范围依据展示（本次用了什么数据源/表/时间范围）
- 问题理解依据（系统识别出什么样的指标、维度、筛选条件）
- 执行摘要（业务语言说明分析思路，失败时结构化失败原因和建议）

### 2.3 🟡 基础设施未打透：分析主题驱动的范围治理

**当前状态**：分析主题存在，但更像"组织入口"，不是"强约束对象"

**需要补的**：
- 多候选数据集/分析对象命中时，缺稳定确认机制
- 会话中当前作用域不显式展示
- 主题从"配置对象"升级为"问数主入口"

### 2.4 🟡 链路不完整：连续分析追问

**当前状态**：有聊天连续性（历史记录），缺少分析连续性

**需要补的**：
- 4 类分析动作产品化：改时间范围、改指标、改维度、加/减筛选条件
- 继承提示（本轮追问继承了哪些上下文）
- 结果绑定推荐追问（基于当前结果生成，非通用问题池）

### 2.5 🟢 P1 增强项

| 能力 | 现状 | 优先级 |
|------|------|--------|
| 预测能力产品化 | 预测模板存在，前端有入口但闭环不足 | P1 |
| 归因/异常诊断 | 较弱 | P1 |
| 问数运营与排障闭环 | 坏 case 回流、执行链路追踪缺失 | P1 |
| 知识治理增强 | 术语/SQL 示例配置已统一，但治理体验待提升 | P1 |

## 4. 结论排序

按用户感知影响和实现可行性排序：

1. **澄清式问答（P0-1）**：最大产品缺口，做不做直接决定用户信任
2. **结果可验证性（P0-1）**：最强感知短板，直接影响是否愿意继续追问
3. **分析主题驱动的范围治理（P0-2）**：问数准确率的前提
4. **围绕结果的连续分析追问（P0-2）**：从一次回答到持续分析的关键

## 5. P0 调整（基于 Quick BI "透明可审计" 设计理念）

Quick BI 不做显式反向澄清（弹窗反问），而是让用户看到推理过程。调整后优先级：

| 顺序 | 能力 | 设计方向 |
|------|------|---------|
| ① | 结果可验证性（推理显性化） | 结果卡片加"问题理解摘要" + 执行详情加业务解释 |
| ② | 分析主题驱动的范围治理 | 前提保障 |
| ③ | 连续分析追问（含连续推荐） | 链路完整 |
| ④ | 知识库增强消歧 | 隐性治理 |

### 5.1 ① 结果可验证性 — 具体设计

**当前已有**：`ExecutionDetailsDrawer.vue`（371行），展示技术执行步骤（匹配术语/SQL示例/表/生成SQL/执行/图表）、耗时、Token数。

**缺的两层**：

1. **结果卡片上的一行"问题理解摘要"**（用户不用点开抽屉就能看到）：
   ```
   系统识别：时间=本月 | 指标=销售额 | 维度=区域 | 数据源=销售数据集
   ```

2. **详情抽屉里的"业务解释层"**，将技术步骤翻译为业务语言：
   ```
   匹配术语 → "识别到业务术语'大客户'，对应 customer_level='A'"
   匹配 SQL 示例 → "基于相似的问法'各渠道月度销售'生成查询"
   选择数据表 → "使用销售数据集 (sales_db.orders)"
   生成 SQL → 展示实际 SQL
   执行 SQL → "查询返回 532 行数据，耗时 0.8s"
   ```

### 5.2 P0 ② — 分析主题驱动的范围治理

**当前状态**：已有分析主题、数据集/文件问数、仪表盘上下文问数骨架，但未形成强约束。

**设计目标**：把"分析主题"从配置对象升级为问数主入口，让问数准确率建立在稳定上下文上。

**具体设计**：

1. **会话范围显式化** — 在聊天界面持续展示当前作用域：
   ```
   ┌──────────────────────────────────────┐
   │ 🎯 当前范围：销售分析主题              │
   │    数据集：sales_db | 默认时间：本月    │
   └──────────────────────────────────────┘
   ```

2. **候选对象确认** — 当多个候选分析对象命中时，不让模型硬猜，而是让用户选择：
   - 如果问题"包含"多个可能的数据集/表，在回答前先让用户明确选择
   - Quick BI 的做法是"分析主题"绑定默认数据集，减少歧义

3. **主题强绑定** — 每个分析主题必须绑定默认数据集、默认时间范围、推荐问题、欢迎语、可选仪表盘绑定

**前端改动**：`useSqlbotNewConversation.ts` 增加当前范围状态；`ChatEntry.vue` 增加范围展示条

### 5.3 P0 ③ — 连续分析追问

**当前状态**：有聊天连续性（历史记录），缺少分析连续性。

**设计目标**：把聊天从"继续聊"升级为"继续分析"。

**具体设计**：

1. **4 类分析动作快捷按钮**（附在每次回答下方）：
   ```
   [改时间范围] [改指标] [改维度] [加筛选条件]
   ```
   点击后不重新开始对话，而是继承当前上下文做修改。

2. **继承提示** — 追问时明确展示继承了哪些上下文：
   ```
   📎 基于上一轮的上下文：时间=本月, 维度=区域
   ```

3. **结果绑定推荐追问** — 基于当前结果生成下一步常见分析动作，而非通用问题池：
   - 当前分析的是"区域销售额"，推荐"按时间趋势""Top 区域""区域对比"

**前端改动**：`SqlbotNewExecutionMetaCard.vue` 增加快捷按钮行；`useSqlbotNewConversation.ts` 增加上下文继承逻辑

### 5.4 P0 ④ — 知识库增强消歧

**当前状态**：术语配置、SQL 示例库已迁移到统一页面，但语义匹配精度依赖 LLM 能力。

**设计目标**：用更强的 RAG 和语义匹配来消歧，减少对 LLM 的依赖，对标 Quick BI 的"轻量知识注入"。

**具体设计**：

1. **语义匹配增强** — 当前术语匹配是简单的关键词匹配，需要增加向量语义搜索
2. **历史消歧复用** — 当用户在对话中澄清了某个歧义（如选择了具体的数据集），将结果记录到"消歧历史"，后续同类问题自动应用
3. **知识优先级** — 知识库命中结果 > LLM 推断结果，当知识库有明确映射时不需要 LLM 再做推断

### 5.5 P1 增强项

| 能力 | 现状 | 设计方向 | 优先级 |
|------|------|---------|--------|
| ⑤ 预测产品化 | 预测模板存在，前端有入口但注释"暂不支持" | 解锁预测功能、实现预测触发/结果呈现闭环 | P1 |
| ⑥ 归因/异常诊断 | 解读已有，归因较弱 | 异常解释、原因拆解、对比归因 | P1 |
| ⑦ 问数运营排障 | 坏 case 回流、执行链路追踪缺失 | 问题反馈闭环、内部运营看板、效果回流 | P1 |
| ⑧ 知识治理增强 | 术语/SQL示例配置已统一迁移 | 批量管理、版本管理、效果统计、导入导出模板 | P1 |

## 6. 完整实施路线

### Phase 1：信任基座（P0 ①②）
- ① 推理显性化（推理透明 + 问题理解摘要 + 业务解释层）
- ② 分析主题范围治理（会话范围显式化 + 候选确认）

### Phase 2：分析链路（P0 ③④）
- ③ 连续分析追问（快捷按钮 + 上下文继承 + 结果绑定推荐）
- ④ 知识库增强消歧（语义匹配 + 消歧复用）

### Phase 3：增强闭环（P1）
- ⑤ 预测产品化
- ⑥ 归因/异常诊断
- ⑦ 问数运营排障
- ⑧ 知识治理增强

## 7. 一句话总结

**问数能力提升路线**：先让用户看得懂（推理透明），再让范围定得准（主题治理），然后让分析连得上（连续追问），最后用知识库做消歧（语义增强），P1 再补预测、归因和运营闭环。
